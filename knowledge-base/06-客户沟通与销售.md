# ClaudeCode 的 172 个应用场景（6）：客户沟通与销售


前面几篇讲了内容、素材、音视频、财务数据和文档处理。

这一篇讲第六大类：客户沟通与销售。

这个模块是我用 Claude Code 很深的一类场景。因为做企业 AI 服务，真正难的不是“会不会做一个 Demo”，而是客户从哪里来，怎么判断客户有没有价值，第一次见面问什么，听完需求后怎么拆方案，方案怎么报价，客户想太多时怎么纠偏，客户不懂时怎么帮他向上汇报，项目黄了以后怎么复盘。

这些事情以前更多靠经验和记忆。现在我会把客户沟通录音、聊天截图、会议纪要、需求文档、客户公司资料、历史案例库都交给 Claude Code，让它帮我做整理、判断、话术、报价和复盘。

这一篇一共 26 个场景。

## 1. 客户沟通录音转写 → 结构化沟通总结 + CRM 建档

这是客户沟通里最基础、也最重要的一个场景。

原来每次和客户、合伙人电话或线下沟通完，如果不马上整理，很多细节很快就忘了。客户说过什么、预算有没有提、谁有决策权、对方到底痛不痛、这条线索值不值得继续跟，都只能靠记忆。时间一长，客户多了，就更乱。

现在我会把录音或转写文本交给 Claude Code。

输入是：一个 m4a、mp4、qta 录音，或者一大段语音转文字。

输出是两部分。

第一部分是完整转写稿。Claude Code 会先用 `ffmpeg` 抽音频，再用 `doubao-asr` 转文字，最后保存成可检索的转写文件。

第二部分是按模板整理出的《客户沟通总结》。里面包括客户信息、业务痛点、需求概览、核心功能拆解、预算工期、产能、销售机会判断、成交概率、客户画像、红黄绿灯风险和下一步待办。

更进一步，它还会把这些资料归档到 Obsidian 的 `客户沟通/` 目录下，按“沟通中 / 已成交 / 已流失”等阶段建立客户文件夹，把总结、转写原文、原始录音放到同一个项目里。

这件事省心的地方是：客户沟通不再散落在录音、微信、脑子里，而是直接进入客户管理体系。

它还能显性化一些关键信号，比如客户有没有预算、当前对接人有没有决策权、客户对报价是否敏感、真实意图是什么、信任度怎么样。对我来说，这就相当于一个轻量 CRM。

## 2. 客户沟通文件夹归类整理

客户资料一多，归档本身也会变成问题。

原来我的 Obsidian 里会散落很多文件：一个客户的沟通总结、同名录音、报价材料、需求说明，有时不在同一个文件夹，命名也不统一。时间久了，再找客户资料就很麻烦。

现在我会让 Claude Code 直接整理客户沟通库。

输入是：口头指令，比如“把这两个文件建文件夹移进去”“时间也加上”“按同样方法处理”“放进沟通中”。

输出是：一批按 `2026-05 项目名/` 规整好的客户文件夹。文件夹里有统一日期前缀，相关录音和文档放在一起，并按客户状态移动到“沟通中”“已成交”“已流失”等目录。

Claude Code 会先 `ls` 看现有目录结构，再 `mkdir` 建文件夹，用 `mv` 移动和重命名文件。关键是它能把我说过的整理模式横向复用到多个项目。

这件事看起来只是文件整理，但实际很重要。销售跟进最怕信息散乱。每个客户一个文件夹、每个阶段状态清楚，后面复盘、报价、写方案、接着沟通，都不会丢上下文。

## 3. 客户/项目需求理解、方案草案与功能拆解

售前阶段，客户的需求通常不是一开始就清楚的。

原来客户可能只会说：“我们想做一个智能体”“我们想监测社交媒体”“我们想做点餐推荐”“我们想让流程更智能”。这些话听起来有方向，但还不能直接报价，也不能直接开发。

现在我会让 Claude Code 把口头需求翻译成方案草案。

输入是：客户需求要点、沟通总结、已有材料，可能是社交媒体监测、点餐推荐、制造业流程、眼镜视光等不同类型需求。

输出是：一份方案草案。里面会包括功能分层拆解，比如采集层、存储层、分析层、展示层；也会有技术架构、差异化能力清单、输入输出逻辑、运行流程，以及按部门或环节拆出来的 AI 场景清单。

这件事省心的地方是：它把客户模糊的口头需求，变成了可以让客户确认的功能清单和架构。

售前很多时候不能一上来谈价格。先要帮客户把“我想要什么”说清楚。客户认了方案，再谈钱才有依据。否则报价容易变成拍脑袋。

## 4. 多档报价方案设计

客户需求拆清楚之后，下一步就是报价。

原来报价最容易纠结。报低了，显得不专业，也会压缩交付空间；报高了，客户可能直接跑；只报一个价，客户没有选择，也不好谈判。

现在我会让 Claude Code 根据功能模块和工作量，设计多档报价。

输入是：已拆解的功能模块、同行报价情报、交付范围、内部成本、风险点。

输出是：基础版、进阶版、全功能版等多档报价方案；工期拆解表；付款节奏，比如 30/40/30；对客户的话术；内部底价、议价空间、红灯风险；以及不包含的费用项，比如云服务器、API、Token 成本。

这个场景里有一个重点：对外报价和内部判断要分开。

给客户看的部分，要讲价值、范围、成果、为什么这个价格合理。内部看的部分，要算底价、风险、工时、哪些需求不能轻易答应、哪些地方会亏。

省心的地方在于：报价不再只是一个数字，而是一套销售结构。三档报价让客户有选择，内部底价让我有边界，付款节奏能降低风险。

## 5. 方案路径判断与客户预期纠偏

很多客户一听 AI，就会想“我要搭智能体”。

但并不是所有需求都需要搭智能体。有些需求用 Claude Code、知识库、Skill、自动化脚本就能解决；有些需求确实需要系统化智能体；有些需求客户已经外包给别人了，不适合重做。

原来如果客户说要做智能体，我很容易顺着他的表达继续聊。但这样可能会推过重、过贵、过复杂的方案，反而增加成交难度和交付风险。

现在我会让 Claude Code 帮我逐个判断：哪些该搭智能体，哪些用更轻的方案更合适。

输入是：客户全部业务需求，比如视频矩阵、营销中台、供应链、职能部门等。

输出是：分类建议。比如视频矩阵可以用 Claude Code + Obsidian + 知识库 + Skill，不一定要搭智能体；供应链对客订单可能更适合做智能体；营销中台如果已经外包，就不要重做；有些地方更适合先做顾问合作。

这件事省心的地方是：它帮我避免为了卖项目而过度设计。

有时候专业感不来自“什么都做”，而来自告诉客户“这个不用做那么重”。用更低成本的方式先建立信任，反而更容易打开后续合作。

## 6. 客户沟通策略、信号解读与销售判断

客户沟通后，我经常会让 Claude Code 帮我判断这条线索。

原来我也能凭感觉判断客户质量，但容易受当时情绪影响。聊得很顺，就容易高估；客户说得热情，也不代表有预算；客户说“50 万、100 万都没关系”，也可能是在释放价格空间。

现在我会把客户身份信息和沟通内容丢给 Claude Code，让它做销售判断。

输入是：客户背景、沟通内容、客户说过的关键话、我的业务定位。

输出是：客户画像、信号解读、报价空间、风险点、下一步策略。它会结合“先培训 → 转顾问 → 转 Agent 定制”的漏斗，帮我拆分阶段打法。

比如客户主动说“50 万、100 万都没关系”，这可能不是客户预算无限，而是说明我原来的定价太低，或者客户期待的是更大价值的东西。Claude Code 会提醒我不要被低价锚定，也不要一开始就把项目做小。

这件事省心的地方是：它把一次销售机会拆成可执行的分阶段打法。不是简单判断“能不能成”，而是判断下一步怎么接、先卖什么、怎么扩大价值、哪些风险要避开。

## 7. 客户沟通信息清单

见新客户前，最怕聊得很热闹，但关键问题没问。

原来我可能会凭经验问客户需求、预算、时间，但不同客户情况不同，临场容易漏。有时候聊完才发现：谁决策没问清，预算没问清，现有系统没问清，时间要求没问清。

现在我会让 Claude Code 先给我一份沟通信息清单。

输入很简单：我要和一个新客户沟通，以及客户大概背景。

输出是：结构化提问清单，包括预算、决策链、当前现状、已有尝试、系统环境、时间要求、内部推动人、成功标准、数据权限、交付边界等。

这件事省心的地方是：把销售沟通标准化，避免漏问关键信息。

它不需要写得很复杂，但非常实用。尤其是新项目、新行业、新客户，先拿一张清单，会让沟通更稳。

## 8. 渠道合作 / B 端伙伴沟通前谈话框架梳理

有时候见的不是客户，而是有 B 端资源的渠道伙伴。

这种沟通和客户沟通不一样。客户关注的是他自己的问题，渠道伙伴关注的是资源、分润、合作模式、客单价、回款周期和谁来交付。

原来临时见这种人，如果没有准备，很容易被对方带节奏，或者一上来就讲方案，结果没摸清对方手里到底有什么资源。

现在我会让 Claude Code 帮我准备谈话框架。

输入可能只是一句话：待会儿要和一个有 B 端渠道资源的人沟通。

输出是一份四块结构化沟通框架：摸对方资源底牌、试探合作契合点、验证可行性、判断优先级。它还会给一些临场建议，比如先问行业、客户规模、决策链、合作模式、客单价、回款、分润比例、对方能触达到什么层级。

这件事省心的地方是：把“待会儿聊聊”变成结构化谈判清单。

第一次沟通的目标不是马上给方案，而是判断这个人值不值得继续聊。这个定位很重要。

## 9. 电话谈判前话术准备 + 谈判复盘

有些电话是关键电话，比如和客户决策人谈时长、价格、交付边界。

原来这种电话只能靠临场发挥。临场一紧张，容易解释太多、让步太快、主动谈价格，或者没有把价值讲透。

现在我会让 Claude Code 先帮我准备话术。

输入是：谈判目标、对象、客户背景、我的底线、需要争取的结果。

输出是：分场景话术。比如对内可以交底，对外只讲价值；要解开客户觉得“太难”的误会；要守住底线；要主动打数据安全牌；价格不要主动往下谈。

电话结束后，我还会把结果丢给 Claude Code 做复盘。

比如客户原本想砍时长，但我把价值讲深之后，反而把半天课做成了一天半。Claude Code 会把这个经验沉淀下来：不要急着解释难度，要围绕客户议题讲价值；客户砍时长时，不一定要防御，也可以把问题讲透，让客户意识到原方案不够。

这件事省心的地方是：谈判经验不再只是一次性的临场发挥，而是可以沉淀成可复用框架。

## 10. 优雅拒绝投资入股的话术设计

有些客户会提出投资入股。

这种话题很敏感。直接拒绝，可能伤关系；不拒绝，又可能给对方错误期待。

原来这种场景很难回。尤其是当客户本身有资源、有诚意，但我并不想接受投资时，需要既守住边界，又不让对方觉得被否定。

现在我会让 Claude Code 帮我设计拒绝话术。

输入是：客户提出投资意向，以及我对这件事的判断。

输出是：一段可直接发的回复，以及背后的判断理由。

核心表达不是“我不要你的钱”，而是把“现在不要”包装成“先做出结果再聊，这是对你负责”。也就是说，不否定未来合作，但现在不把股权和资本关系提前绑死。

这个场景背后的判断是：我的业务目前更接近轻成本模型，如果过早引入资本，可能和资本预期冲突。客户希望投资，未必真的理解这个业务的交付方式和增长节奏。

这件事省心的地方是：高敏感场景有现成话术，既守住业务边界，也维护关系。

## 11. 高管 AI 私教产品线设计

这个场景来自高管 AI 私教。

原来有人来约 AI 私教，容易把它理解成培训、咨询或陪跑。但这三者其实不同。高管私教卖的不是课时，也不只是讲工具，而是帮助高管形成 AI 认知、个人工作流和长期使用习惯。

现在我会让 Claude Code 帮我把它设计成一条产品线。

输入是：沟通背景、目标高管画像、我自己写的框架草稿。

输出是：完整产品定义。包括定位、交付物、定价、交付边界、6 个月路径、数据合规红线、续费和转介绍设计。

比如这条产品线可以定义为：按周期和结果卖，而不是按课时卖。交付物包括线下深度课、微信答疑、AI AS ME 资产包。定价可以卡在 CFO 可直接审批的价格区间，比如 1.5 万/月。

Claude Code 还会帮我做多轮打磨：私教不等于培训，也不等于普通陪跑；频次不要设计太密；可选项不要太多；场景要排优先级；资产沉淀要单独算交付物；边界条款要防止无限白嫖。

这件事省心的地方是：它把一个模糊的新机会，设计成一个可卖、可交付、可续费的产品。

## 12. 朋友的创业项目 / BP 商业评估

有时候朋友会拿一份 BP 来问我怎么看。

原来我可以凭直觉点评，但很容易停留在泛泛建议，比如市场大不大、团队行不行、技术有没有壁垒。

现在我会把 BP PDF 交给 Claude Code，让它结合我的背景做评估。

输入是：一份创业项目 BP，比如海外短剧内置情景广告项目。

输出是：市场判断、团队可信度、产品逻辑、真正护城河、技术指标是否可信、协同点、需要追问的问题。

这个场景里，Claude Code 不只是总结 BP 内容，而是结合我的资源和业务背景判断。比如短剧广告项目，真正护城河可能不是 AI，而是短剧库存、渠道和商业化资源；技术指标如果没有验证，就不能只看 BP 上写得好。

这件事省心的地方是：它能帮我快速吃透一份 BP，并给出更有商业判断力的评估。

## 13. 客户向上汇报材料代写

很多时候，客户对接人听懂了，但他没法向老板讲清楚。

这时候项目能不能继续，不只取决于我讲得好不好，也取决于客户内部能不能汇报得清楚。

原来我可能只是把方案发给对接人，让他自己去汇报。但对方未必能讲清楚为什么要做、收益是什么、预算为什么合理、前期方案为什么值得投入。

现在我会帮客户写向上汇报材料。

输入是：多场会议的沟通总结、客户需求、报价方案、合作模式。

输出是：写进飞书的汇报报告。内容包括为什么做、可量化收益、报价方案、合作模式、前期方案评估，以及老板需要关心的重点。

这类材料要比方案书更短、更重点。Claude Code 可以按“简短重点”“删模块”“突出收益”不断迭代。

这件事省心的地方是：它帮客户对接人过内部决策这道坎。

很多 B 端项目不是客户没兴趣，而是内部汇报讲不清。帮客户写好向上材料，本质上也是销售推进的一部分。

## 14. 跨客户项目匹配 + 真实案例与客户需求精准映射

客户经常会问：“你们有没有做过类似的？”

原来回答这个问题很依赖记忆。我做过很多项目，但要在当下快速找到和客户需求最匹配的案例，并且讲出对应关系，并不容易。

现在我会让 Claude Code 从历史客户和案例库里找匹配案例。

输入是：新客户需求，比如制造业 RAG、文本配置识别、供应商评分、流程审核。

输出是：按匹配度排序的案例映射。比如文本配置识别可以对应立讯 ISO 审核项目；供应商评分可以对应过去的监控预警或多 Agent 经验。每个映射还会有星级、可讲金句、通用方法论和销售建议。

这个场景的关键是“精准映射”。

不是随便说“我们做过知识库”，而是要把客户需求拆成能力点，再用历史项目里的具体能力去对应。这样客户才会觉得你真的做过类似的，而不是泛泛吹经验。

这件事省心的地方是：历史项目档案变成了可调用的案例库。每次新需求来了，都可以快速找到最有说服力的对标案例。

## 15. 客户案例材料撰写 + 脱敏 + 按需扩写

有了真实项目，还要把它写成可对外发送的案例材料。

原来内部项目复盘往往不能直接发给客户。里面可能有真实公司名、地区、业务细节、内部文档、敏感数据。如果直接发，有泄密风险；如果脱敏太狠，又没有说服力。

现在我会让 Claude Code 帮我写对外案例，并做分步脱敏。

输入是：相关飞书文档、Obsidian 项目文档、目标飞书文档、脱敏要求。

输出是：结构化案例材料，包括背景、痛点、方案、技术架构、知识库治理、价值、扩展方向。脱敏会分几步做：先去公司名，再去地区，再去具体业务，但保留“做了什么”和关键数字。

这件事省心的地方是：内部项目可以转成销售材料，而且不会因为脱敏而失去说服力。

比如制造业客户很吃“数据详实”这套。只说“我们做过一个项目”没用，要能讲清楚业务背景、文件规模、流程复杂度、解决了什么问题。但又不能暴露客户是谁。Claude Code 可以在这之间做平衡。

## 16. 项目复盘与经验沉淀

每个项目结束后，不管成交还是流失，都应该复盘。

原来项目黄了就黄了，最多口头说一句“下次注意”。但如果不沉淀经验，很容易下次重复踩坑。

现在我会把项目全部资料交给 Claude Code 做复盘。

输入是：项目沟通记录、报价、合同、客户反馈、真实成本、最后结果。

输出是：复盘结论，以及写入经验库或教训库的条目。比如“报价用打包价，不用单价锚”“体制内客户报价前先问预算”“合同主体未必是采购方”“缺前期销售感”“不要低估客户内部汇报难度”。

如果项目流失，还会把客户文件夹移动到“已流失”，并标注流失原因。

这件事省心的地方是：每个项目都能变成复利资产。

成交项目沉淀案例，流失项目沉淀教训。销售能力不是靠一次次感觉提升，而是靠一次次复盘积累判断。

## 17. 见客户前公司 / 创始人深度尽调

见客户前，尤其是见董事长、老板、创始人，不能空手去。

原来我可能会简单搜一下公司官网、百度百科，看个大概。但如果要做到“懂行”，这些远远不够。你要知道公司做什么，老板是谁，最近战略是什么，行业位置如何，公开报道里讲过什么，进门才能聊到对方关心的点。

现在我会让 Claude Code 做会前尽调。

输入是：公司名、起点链接、调研维度。

输出是一份结构化报告。包括公司核心事实表、创始人或董事长画像、公开金句、当前战略、重要新闻、以及 5 个会面切入点。每个切入点会区分公开事实和推断。

这个场景里，Claude Code 会通过官网、政府报道、媒体报道、公开资料等一手来源做交叉验证。如果信息少，也会明确说查不到，不会硬编。

这件事省心的地方是：空白客户可以在几小时内变成可对话对象。

更重要的是，它能避免讲错。比如客户到底是食品工业还是餐饮连锁，方向一错，进门就露怯。会前尽调可以帮我先纠偏。

## 18. 行业痛点与 AI 可解 / 不可解判断

只了解客户公司还不够，还要了解客户所在行业。

原来见某个行业客户前，我可能知道一点行业常识，但很难快速掌握行业痛点、数据、趋势和 AI 切入点。

现在我会让 Claude Code 做行业痛点调研。

输入是：行业名称和调研维度。比如食品行业，可以看市场规模、Z 世代、健康化、年货季节性、即时零售、抖音电商、用工、新品成功率、种草 ROI、国潮等。

输出是：痛点 Top 10 表，每条带数据；每条痛点 AI 能不能解决的判断；3 个最适合切入的场景；3 个看似能做但其实不适合的场景。

这件事省心的地方是：我不是拿着一堆行业资料去见客户，而是拿着“该推什么、别推什么”的判断去见客户。

做企业 AI 服务，最怕什么都说 AI 能解决。客户真正需要的是你能判断边界：哪些值得做，哪些不值得做，哪些先做，哪些以后再说。

## 19. 竞品 / 同行业 AI 落地案例弹药库收集

见客户时，讲道理不如讲同行案例。

原来我可能知道行业里有人做 AI，但没有具体公司名、动作、数据和来源。客户听起来就觉得虚。

现在我会让 Claude Code 收集同行 AI 落地案例。

输入是：对标品牌清单和 AI 场景类别。

输出是：“品牌 × AI 应用 × 动作 × 数据 × 来源”的案例矩阵，以及每类 1 到 2 个有名有姓的案例。还可以整理成 30 秒案例卡片，方便现场讲。

比如某个食品品牌用 AI 缩短新品周期，某个企业用 AI 质检提升准确率，某个品牌用 AI 做电商内容生产。每个案例都要有公司名、动作、数据和来源。

这件事省心的地方是：它给销售准备了硬弹药。

尤其是面对董事长或老板，抽象讲 AI 没用。更有效的是告诉他：同行已经怎么做了，不做可能会被拉开差距。但这些案例必须有证据，不能编。

## 20. 董事长级首次拜访沟通方案设计

见董事长和见执行层，不是一套打法。

原来如果按普通方案沟通，很容易一上来就讲案例、讲功能、讲我们能做什么。但董事长不一定关心功能细节，他更关心战略、增长、风险、组织、钱和人。

现在我会让 Claude Code 帮我设计董事长级首次拜访方案。

输入是：拜访前沟通记录、客户公司信息、行业调研、对方角色。

输出是：董事长和执行层差异表，以及一套升级版五步法。比如先闭嘴让他说，再用行业观察对位焦虑，再抛董事长级问题，再画地图不给方案，最后收尾留钩子。

它还会给开场白、临场反应表、会前准备清单。

这件事省心的地方是：把“急于证明能做事”改成“先证明懂行”。

面对董事长，太早抛案例、姿态像顾问、急着卖方案，都可能不合适。更好的打法是先让对方感受到你理解他的行业和问题，再逐步引导到合作空间。

## 21. 客户拜访后语音反馈复盘 + 打法校准

见完客户之后，也需要复盘。

原来拜访后，销售伙伴可能会发一段语音，说今天聊得怎么样、对方什么反应、有哪些新信息。如果不及时整理，这些反馈很快就散掉。

现在我会把拜访后语音转写，再让 Claude Code 做复盘。

输入是：拜访后语音转写。

输出是：整体结果、之前假设的校准、对方画像、核心诊断、暴露出来的关键信息、合作模式、下次要做的三件事、时间窗口。

比如会前以为见的是二代，实际见的是一代；原来准备的话术偏执行层，实际对方只听得懂结果、案例和分钱。Claude Code 会根据反馈调整下一次打法。

这件事省心的地方是：每轮客户拜访都形成“反馈 → 校准 → 下次方案”的闭环。

销售不是一次性表达，而是连续校准。见一次客户，就要更新一次判断。

## 22. 双人配合销售脚本设计

有时候不是我一个人见客户，而是和销售搭档一起去。

原来双人见客户容易乱。谁开场、谁讲行业、谁讲案例、谁接问题、谁收尾，如果没有提前分工，很容易抢话、漏话，甚至搭档讲了不该讲的案例。

现在我会让 Claude Code 设计双人配合脚本。

输入是：两个人的角色关系、案例归属、客户背景、见面目标。

输出是：角色定位表和六阶段分工脚本。比如开场、行业大盘、切到本人案例、算账映射、留白指人、收尾约下次。还会给搭档能讲什么、不能讲什么，以及风险应对。

这个场景里很重要的一点是案例归属。如果所有案例都是我亲手做的，销售搭档不能自己讲得像亲历者，否则很容易露馅。更好的方式是设计过桥句，比如“这块小曾是亲手做的，让他给您讲讲”。

这件事省心的地方是：双人销售不再靠默契，而是有脚本配合。看起来更像成熟团队，也能避免现场翻车。

## 23. 降本场景筛选与第一票合作模式设计

客户给了试点入口之后，关键是选第一个场景。

原来客户可能会说“你们看哪里能做”，这时候如果选错场景，第一单就容易翻车。场景太难，做不出来；场景太小，客户没感觉；场景太敏感，内部推进困难。

现在我会让 Claude Code 帮我筛选第一票合作场景。

输入是：客户诉求，比如“要明显感觉到 AI 价值，而且不能翻车”。

输出是：候选场景的三维打分表，通常按翻车代价、见效速度、价值规模来评估。然后给出最优组合、现场二选一话术和合作模式设计。

比如主公司财务可能是“王炸”场景，子公司内容冷启动也不错，但质检、客服可能不适合作为第一票。合作模式可以设计成三阶段：试点先做后收、按项目制、年度顾问费加分成。甚至可以设计低风险体验包，比如做不出来退一部分钱。

这件事省心的地方是：把模糊的“怎么合作”落到客户可以点头的颗粒度。

第一票的目标不是把所有东西做完，而是选一个安全、可见效、能建立信任的场景。

## 24. 项目接单评估

不是每个项目都应该接。

原来客户一来咨询，容易觉得都是机会。但有些客户预算低、需求乱、决策链长、期望不合理，接了可能亏时间。

现在我会让 Claude Code 根据客户沟通记录做接单评估。

输入是：客户沟通项目 Markdown，比如某个养老行业标书智能体项目。

输出是：客户基本盘、团队规模、月单量、预算、是否问过几家、对我沟通表现的评分、正负面信号清单和接单建议。

比如它可以评价：我专业表现 9/10，但销售表现 6/10；客户有真实需求，但预算和决策链不明；项目值得继续聊，但不宜投入太多免费方案时间。

这件事省心的地方是：客户沟通记录变成了接单决策依据。

它也能帮我复盘自己的销售表现。不是只判断客户行不行，也判断我哪里讲得好、哪里讲得弱。

## 25. 微信聊天截图 / 合伙人沟通解读

很多业务信息来自微信截图、群聊、私聊或合伙人语音。

原来这些信息很碎。对方发几张图、几句话，我要自己判断他到底什么意思，是不是跑偏了，要不要回，怎么回。

现在我会把截图或语音转写交给 Claude Code 解读。

输入是：多张聊天截图、项目图、合伙人语音转写，必要时再加 Obsidian 历史记录。

输出是：对方意图解读、是否跑偏的判断、下一步建议。如果是久未联系的合伙人，它还可以帮我从历史记录里恢复这段关系的上下文。

这件事省心的地方是：碎片化聊天信息可以被整理成清晰局势判断。

有些时候，别人发来的不是“需求”，而是一堆信号。Claude Code 可以帮我从这些信号里判断：对方是想合作，想甩锅，想试探，还是只是随便问问。

## 26. 是否回复 / 如何回复消息的社交决策

最后这个场景很小，但很真实。

有些人发来消息，你知道从他身上赚不到钱，也不太可能产生长期价值。回吧，消耗精力；不回吧，又怕关系太僵。

原来这种事也会占用脑子。虽然不大，但会反复打断。

现在我会直接问 Claude Code：这种消息要不要回，怎么回。

输入是：对方是谁、对方价值、消息内容、我纠结的点。

输出是：回复策略。可能是简短回一下，也可能是不回，也可能是礼貌但不展开。

这件事省心的地方是：它帮我做社交取舍，把精力留给更高价值的关系。

销售和商务里，不是所有关系都值得投入同样精力。学会判断哪些该认真经营，哪些简单处理，也是效率的一部分。

## 这类场景真正解决了什么问题

客户沟通与销售这一类场景，本质上解决的是一件事：把销售从“靠感觉”变成“有记录、有判断、有打法、有复盘”。

客户沟通不是聊完就结束。每一次沟通，都应该沉淀成客户档案；每一次需求，都应该拆成方案和报价；每一次谈判，都应该沉淀话术和经验；每一次拜访，都应该校准下一步打法；每一个流失项目，也应该沉淀成教训。

Claude Code 在这里的价值，不是替我卖东西，而是帮我把销售过程结构化。

录音变成 CRM。  
需求变成方案。  
功能变成报价。  
客户话语变成信号。  
历史项目变成案例库。  
拜访反馈变成下次打法。  
流失项目变成经验库。

这里有几个关键原则。

第一，客户信息必须归档。只靠微信、录音和记忆，客户一多一定乱。每个客户都应该有文件夹，有状态，有沟通记录，有下一步动作。

第二，销售判断要基于信号。客户说了什么、没说什么、谁有决策权、预算是否明确、是否愿意推进，这些都要提取出来，而不是凭感觉判断。

第三，方案和报价要分层。不要一上来只给一个价格，也不要让客户用低价锚定你。需求拆清楚之后，报价要有版本、有边界、有付款节奏、有内部底价。

第四，案例库是销售资产。做过的项目，如果不能在新客户面前快速调用，就没有发挥最大价值。历史项目要能脱敏、映射、复用。

第五，每次项目都要复盘。成交了沉淀案例，流失了沉淀教训。销售能力真正提升，靠的是一次次把经验写下来，而不是靠模糊感觉。

## 下一篇写什么

这篇讲的是第六大类：客户沟通与销售。

下一篇可以继续拆第七大类：项目方案与交付。

如果说客户沟通与销售解决的是“怎么把客户机会推进成项目”，那项目方案与交付解决的就是“项目来了以后，怎么把需求翻译成方案、MVP、系统设计、交付路径和可执行任务”。
