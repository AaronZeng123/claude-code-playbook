# ClaudeCode 的 172 个应用场景（9）：调研与情报


前面一篇讲的是企业培训。

这一篇讲第九大类：调研与情报。

这个模块不是单纯“上网搜资料”。它更像是把一个不确定问题，变成可以做判断、做选型、做销售、做方案、做内容的情报。

我现在做很多事情之前，都会先让 Claude Code 做一轮调研：

一个 GitHub 项目到底值不值得装？某个方向有没有现成工具可以借？某个 Skill 或 MCP 要不要装？国内外 MCP 生态到底发展到什么程度？国内 CLI 工具哪些是真的、哪些已经凉了？一个 AI 工具动态是真是假？某家公司什么背景？某个政策我能不能申报？某个政府网站的信息怎么整理成报告？甚至一个审计项目的全流程，我也可以先让 Claude Code 帮我理解清楚。

所以这一类场景解决的不是“找网页”，而是“把信息变成判断”。

这一篇一共 10 个场景。

## 1. GitHub 开源项目研究与批量速读选型

这是我用得很高频的调研场景。

平时会看到很多 GitHub 项目：别人安利的、文章里提到的、工具榜单里看到的、某个 Skill 仓库里找到的。问题是，这些项目看起来都挺有用，但实际上能不能用、值不值得装、能不能变成 Claude Code 的 Skill、和我的业务有没有关系，都需要判断。

原来我只能一个个打开 GitHub，看 README、看 Star、看安装方式、看 issue、看技术栈。少量项目还可以，多了就很耗时间。比如一次要核验 18 个链接、26 个人物 Skill、39 个本地抓取 repo，手工看基本不现实。

现在我会让 Claude Code 批量研究 GitHub 项目。

输入可以是一个 repo URL，也可以是一批链接。比如 open-design、ljg-skills、ai-hedge-fund，或者一批待核验的 GitHub 链接。

输出通常包括几部分：项目是做什么的，Star 数和活跃度，技术栈和架构，安装方式，能不能装成 Claude Skill，对我的业务有没有价值，我的本机硬件能不能跑。如果是批量处理，还会输出一张表格，包括链接是否正确、项目名称、描述、Star、安装命令和一句话定位。

比如有些项目看起来很火，但我的 M5 Air 32GB 根本跑不动；有些项目 Star 不低，但安装复杂、依赖重、和我业务关系弱；有些 Skill 看起来有趣，但其实不如我已有工具稳定。

Claude Code 会用 `gh repo view`、clone、读 README、联网搜索等方式，把项目基本信息先摸清楚。批量任务里，它还会检查链接是否正确，遇到旧地址迁移或 301 跳转，会尝试定位新地址。

这件事省心的地方是：我不用凭一篇安利文就去装工具。

开源项目调研最怕跟风。别人说好用，不代表适合我；Star 高，不代表能落地；能跑，不代表值得跑。Claude Code 可以先做一轮技术尽调，帮我判断“这个东西到底要不要花时间”。

## 2. 特定领域开源项目 / 工具 / API 调研

有时候我不是研究某一个项目，而是想知道某个方向有没有现成轮子。

比如我要做财务工具、招投标报价体系、抖音下载、XMind 编程、长文转 Twitter 或小红书工具。这个时候不应该一上来就自己写代码，而是先看看行业里有没有现成项目、API 或开源工具。

原来我会自己搜关键词，一个个点开项目。问题是搜索结果很多，有些过时，有些不可用，有些 Star 虚高，有些安装很麻烦。花一晚上看完，可能还是不知道该选哪个。

现在我会让 Claude Code 按方向做工具调研。

输入是：一个方向或关键词。比如“财务相关 GitHub 项目”“抖音下载工具”“XMind 编程工具”“长文转社媒工具”。

输出是：分类项目清单。每个项目包括名称、描述、语言、Star、亮点、链接和选型建议。

比如财务方向，它会列出 Firefly III、Actual、Beancount 这类工具，并说明它们分别适合个人记账、预算管理、纯文本复式记账。抖音下载方向，它会对比 douyin-downloader 和 TikTokDownloader，并说明签名机制差异。XMind 方向，它会提醒我：XMind 文件本质上是 ZIP，可以直接改写，也可以考虑 xmind-sdk-js 或 xmind-mcp。

这件事的价值是：先找轮子，再决定要不要造轮子。

很多项目不是技术上不能做，而是没有必要从零做。Claude Code 可以帮我把已有生态摸一遍，给出“直接用”“参考实现”“暂不建议”的判断。

这比自己盲目开干更稳。

## 3. Claude Code Skill / MCP 生态选型调研与对比

除了普通 GitHub 项目，我还经常要调研 Claude Code 的 Skill、MCP、插件生态。

比如我要找 PPT 相关 Skill、泳道图 MCP、财务资料 Skill、Obsidian 工具、微信读书工具、去 AI 味工具、电商相关能力。每一类工具可能都有多个候选项，但安装方式、依赖、是否付费、安全风险都不一样。

原来我会直接装来试。但这样很容易浪费时间。装完才发现要 API Key，要付费，跑不通，或者功能根本不符合预期。

现在我会先让 Claude Code 做选型调研。

输入是：一个能力需求，或者一批具体 Skill / MCP 名称。

输出是：候选清单和对比表。里面会包括安装量、来源仓库、核心能力、安装方式、是否需要 API Key、是否付费、是否有安全风险，以及推荐优先级。

比如我要找泳道图工具，它会对比 AI Charts MCP 和 Draw.io MCP。我要找微信读书工具，它会对比官方 Skill 和基于 Cookie 的 mcp-server-weread。我要找 PPT 工具，它会查哪些 Skill 更成熟，哪些需要额外环境。

Claude Code 会用 `find-skills`、`npx skills find`、skills.sh leaderboard、GitHub README、WebSearch 等方式拿信息，再按安装量、来源可信度、功能匹配度和前置条件筛选。

这件事省心的地方是：装之前先判断，不要装完才后悔。

很多工具不是不能用，而是有前置成本。比如要 API Key、要登录态、要付费、要特定系统环境、要安全授权。如果这些东西不提前看清楚，后面会浪费很多时间。

我现在更倾向于把 Skill / MCP 选型当成一个小型采购流程：先调研，再安装，再实测，不满意就卸载。Claude Code 可以把这个流程标准化。

## 4. MCP 生态全景调研

这个场景比单个 Skill / MCP 选型更大。

有时候我不是想找一个工具，而是想系统摸清国内外 MCP 生态发展到什么程度：哪些大厂已经有官方 MCP，哪些只是社区封装，哪些领域还没有成熟方案，哪些项目已经归档不能再用。

原来这类事情很难靠普通搜索完成。因为 MCP 生态更新快，名字也不统一。有些是官方仓库，有些是社区仓库，有些只是别人写的 demo，有些项目已经停止维护。单看一个 awesome 列表，很容易误判。

现在我会让 Claude Code 做 MCP 生态全景调研。

输入是：分类厂商和应用清单，比如国内大厂、国内消费办公、开发者工具、海外官方四个方向，并要求“不编造，没有就写暂未发现”。

输出是：分领域表格。每个条目包括 MCP 名称、官方还是社区、仓库链接、30 字能力说明，以及汇总规律。

比如它会发现：阿里、火山这类厂商有比较多官方能力集合；飞书、钉钉已经官方下场；微信、小红书、抖音更多是社区方案；海外 GitHub、Notion、Stripe 等有官方或较成熟实现；OpenAI、Discord 这类可能没有第一方 MCP，需要明确标注。

Claude Code 会通过多轮 WebSearch、中英文关键词、`site:github.com`、精确引号匹配、官方组织核验、awesome-mcp-servers、PulseMCP 等来源交叉验证。它还会区分官方、社区、远程 OAuth、已归档项目，并识别网页里夹带的 prompt injection，不照单全收。

这件事省心的地方是：我可以一次性建立一张中外 MCP 地图。

做企业 AI 落地和 Agent 定制时，这很有价值。因为接入能力的成熟度，直接影响方案可行性。官方 MCP、社区 MCP、没有 MCP，三种情况对应的交付风险完全不同。

## 5. 国内命令行工具 CLI 全景深度调研

这个场景和 MCP 类似，但关注的是国内 CLI 工具生态。

有时候我需要知道：国内云厂商、办公协作工具、AI 编程工具、开发者工具、消费类应用，到底有没有官方或社区 CLI？哪些工具真的可用？哪些已经停更？哪些只是传闻？

原来这类信息很散。很多工具名字像真的，但实际搜不到；有些项目几年前还活跃，现在已经凉了；有些命令行工具只支持很窄的场景。

现在我会让 Claude Code 做 CLI 全景调研。

输入是：分类清单，比如云厂商、办公协作、AI 编程、开发者通用、消费类应用四大方向。

输出是：分类调研报告。每个工具会标注状态：存在、不开源、已停更、活跃、明显已凉、不存在。报告还会写关键发现和带链接的来源列表。

比如它会纠正一些误判：某个“小红书 CLI”可能并不存在；某些云厂商 CLI 是官方稳定工具；某些 AI Agent CLI 值得关注；某些消费类应用的 CLI 基本只能靠社区逆向。

这个场景的价值是：先辨别真伪，再谈选型。

做方案时，不能客户一问“能不能命令行操作某某平台”，我就凭感觉说可以。要先知道有没有官方 CLI，有没有社区工具，是否可维护，是否存在账号风险和合规风险。

Claude Code 可以把这些分散信息整理成工具地图，为后续方案选型和客户沟通储备弹药。

## 6. AI 工具动态核实 + 竞品格局对比

AI 工具圈每天都有很多动态、截图、传言。

比如某个工具新增了浏览器插件，某个模型登录页显示注册关闭，某家公司和另一家公司关系变化，推特上突然出现一堆关于某个产品的讨论。

原来看到这些消息，很容易直接转发或写内容。但很多信息未必准确。有些是截图断章取义，有些是二手转述，有些是营销号夸大，有些其实还没有官方确认。

现在我会让 Claude Code 先核实。

输入可以是一段动态文本、一张截图、一个推特链接，或者一个需要核实的问题。

输出是：核实后的事实、时间线、各方立场、竞品对比表，以及多来源链接。

比如 Codex 新增 Chrome 插件，就不能只看一条朋友圈动态。要去找官方信息、用户反馈、截图来源，再和 Claude Code 做竞品对比：Codex 这次增强了什么能力，和 Claude Code 在浏览器操作、代码执行、工作流方面有什么差异。

再比如 xAI 和 Anthropic 的关系，也不能只看某个人的推文。要查多方报道、当事方信息、公开动态，再整理成事件时间线。

Claude Code 会用 WebSearch、推特抓取、图片识别等方式交叉验证。最后给出的不是“网上有人说”，而是“能找到哪些来源，哪些确定，哪些存疑”。

这件事省心的地方是：把朋友圈级传言，变成可追溯的判断。

对做内容的人来说，这非常重要。AI 圈热点来得快，但如果不核实就写，很容易翻车。Claude Code 可以先把真假、来源、竞品格局和可写角度整理出来，再决定要不要写。

## 7. 行业 / 工具 / 公司联网调研

这是更通用的一类调研。

有时候我要了解一个工具品类，比如语音输入法；有时候要了解一家公司，比如客户公司背景；有时候要为内容选题补行业数据，比如电商、程序员、老板、财务这些人群的规模、痛点、AI 渗透率和案例数据。

原来这类调研非常碎。要搜索多个关键词，打开多个网页，判断哪些信息可信，再整理成表格或报告。人工做当然可以，但很耗时间，而且容易漏来源。

现在我会让 Claude Code 做联网调研。

输入可以是一个工具品类、一个公司名、一个目标人群，或者一批文章标题。

输出会根据任务不同而变化。

如果是工具品类，输出可能是两大阵营对比表、Mac 推荐和一句话选型建议。

如果是公司调研，输出可能是结构化公司画像，包括主营业务、规模、发展历史、公开信息、可能需求和沟通切入点。

如果是人群调研，输出可能是每个标题对应的 5 条素材，包括行业规模、痛点、AI 渗透率、真实案例和来源链接，最后写入调研库。

这个场景里，我会特别要求两点：

第一，数据要有来源。没有来源的数据，不要写得像确定事实。

第二，找不到就写未找到。不要为了让报告完整而编造。

这件事省心的地方是：调研结果可以直接进入后续写作、销售或方案，而不是停留在搜索记录里。

比如做内容时，它能给文章提供事实支撑；见客户前，它能让我先理解公司背景；做工具选型时，它能帮我快速判断该用哪个工具。

## 8. 政府扶持政策匹配度逐条分析

这个场景和企业经营相关。

有时候看到一份政府扶持政策，里面有很多条款，比如认定条件、资金支持、办公补贴、模型券、研发补贴、人才政策。看起来都可能有用，但要判断自己到底能不能申请，人工读起来很累。

原来我看政策文件，最容易的问题是：条款很多，但不知道哪些和我有关。比如一人公司能不能申？AI Agent 咨询算不算？Token 支出能不能算模型使用成本？持股结构有没有影响？哪些政策门槛低，哪些只是看起来好但实际上不符合？

现在我会把整份政策原文交给 Claude Code。

输入是：完整政策文件，比如《海淀区打造 OPC 创业生态措施》，再加上我的业务背景：一人公司、AI Agent / 咨询、股权情况、Token 支出、办公情况等。

输出是：逐条对照分析。

它会把政策拆成两块：认定条件和支持政策。然后逐条判断我是否符合，哪些需要确认，哪些不符合。最后整理成可拿补贴清单、按门槛排序的申报优先级，以及还需要补充的信息。

比如某些措施可能有 10 万补贴，某些可能和安居、研发、模型券相关，但不同政策要求不同。Claude Code 会先帮我把“能不能申”和“先申哪个”说清楚。

这件事省心的地方是：一份很长的政策文件，可以变成行动清单。

政策文件不是用来收藏的，关键是判断自己符不符合、能拿什么、先做哪一步。Claude Code 在这里更像一个政策初筛助手。

## 9. 政府 / 资讯网站浏览取数 + 深度调研报告

有些调研不只是读一份政策，而是要从政府网站、资讯网站里查某个主题的最新动态。

比如经开区老旧小区改造最近一个月有什么进展，施工方是谁，投资额多少，工期多长，政策背景是什么。再比如在人人都是产品经理站内搜索 Claude Code，提取前 5 篇文章标题、作者、日期和 URL。

原来这类工作很碎。要打开政府网站，找搜索框，搜关键词，点开多篇文章，复制标题和链接，再整理成报告。政府网站的结构还经常不统一，手工做很耗时间。

现在我会让 Claude Code 用浏览器自动化处理。

输入是：网站链接和主题。比如政府网站 + “老旧小区改造近一月新闻”，或者资讯站 + “Claude Code”。

输出是：结构化调研报告，或者前几篇文章的标题、作者、日期、URL。更完整的任务里，它还会把报告写成 Markdown，存到桌面，再同步到飞书。

Claude Code 会通过 `web-access` 或 `agent-browser` 打开网站，定位搜索框，输入关键词，提交搜索，再从 DOM 里提取结果。如果需要拓展调研，还会结合 WebSearch 和 WebFetch 补充背景信息，并行写报告。

这件事省心的地方是：原本“逛官网找信息”的动作，可以变成结构化调研任务。

尤其是政府网站和资讯网站，信息有价值，但人工浏览很低效。Claude Code 可以帮我把零散网页整理成可读报告。

## 10. 年度审计 / 审计项目全流程理解

最后一个场景比较特殊，它不是工具选型，也不是行业调研，而是咨询型答疑。

有时候我需要快速理解一个专业流程，比如年度审计项目。这个时候不是要查某个新闻，而是要建立完整的业务认知：审计项目从方案、底稿、PBC、证据收集、数据清洗、样本审查、报告到整改跟踪，每一步到底在做什么，有什么坑，怎么改进。

原来这种事情只能靠请教专业人士，或者自己找审计资料慢慢看。短时间内很难形成结构化认知。

现在我会把审计全流程清单交给 Claude Code，让它逐环节拆。

输入是：审计全流程清单，比如方案、底稿、PBC、证据收集、数据清洗、样本审查、报告、整改跟踪。

输出是：每个环节的核心问题、常见坑和改进做法。

比如底稿要有勾稽关系，PBC 要区分“必须提供”和“如有则提供”，证据收集要保证可追溯，数据清洗规则要书面化，样本审查要能解释抽样逻辑，整改跟踪不能只停留在报告结论。

这件事省心的地方是：它能帮我快速建立一个专业流程的认知骨架。

我不一定马上变成审计专家，但至少能知道这个业务怎么流转，客户可能卡在哪里，哪些地方适合 AI 介入，哪些地方必须由专业人员判断。

这类场景对咨询很有用。很多行业项目，前期最缺的不是技术方案，而是对业务流程的基本理解。Claude Code 可以先帮我搭骨架，再让我继续追问细节。

## 这类场景真正解决了什么问题

调研与情报这一类场景，本质上解决的是“判断前的信息质量”问题。

很多决策看起来是判断力问题，其实前面缺的是可靠信息。

要不要装一个 GitHub 项目，要先知道它是什么、怎么装、能不能跑。  
要不要用一个 Skill，要先知道它要不要 Key、是不是付费、有没有安全风险。  
要不要接一个 MCP，要先知道有没有官方实现，社区方案是否可靠。  
要不要相信一个 AI 工具动态，要先核实真假和来源。  
要不要见一个客户，要先知道公司背景和行业痛点。  
要不要申请一个政策，要先逐条对照自己符不符合。  
要不要切入一个专业流程，要先理解这个流程的关键环节和坑。

Claude Code 在这里的价值，不是替我搜索，而是替我把搜索结果变成判断。

它可以读 GitHub。  
可以查 Skill 生态。  
可以画 MCP 地图。  
可以盘 CLI 工具。  
可以核实 AI 工具动态。  
可以做公司画像。  
可以补行业数据。  
可以拆政府政策。  
可以浏览政府网站取数。  
可以快速理解审计这类专业流程。

这里有几个关键原则。

第一，调研不是为了堆资料，而是为了做判断。最终输出一定要有结论：值不值得用，适不适合装，能不能讲，是否可信，下一步怎么做。

第二，来源必须清楚。尤其是行业数据、客户背景、竞品案例、政策条款，不能只写结论。没有来源的数据，用起来会有风险。

第三，找不到就写找不到。调研最怕为了完整而编。Claude Code 应该帮我查证，而不是帮我补脑洞。

第四，调研结果要进入工作流。GitHub 调研要进入工具选型，MCP 调研要进入系统接入判断，行业调研要进入销售拜访，政策调研要进入申报动作，专业流程调研要进入方案设计。

第五，调研要服务边界判断。很多时候专业感不是“什么都能做”，而是知道哪些能做、哪些不能做、哪些值得先做、哪些暂时别碰。

## 下一篇写什么

这篇讲的是第九大类：调研与情报。

下一篇可以继续拆第十大类：知识管理。

如果说调研与情报解决的是“外部信息怎么查清楚”，那知识管理解决的就是“这些信息、经验、案例、规则、素材怎么沉淀下来，并在下一次写作、销售、交付、培训时重新调用”。
