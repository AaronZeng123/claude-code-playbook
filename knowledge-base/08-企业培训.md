# ClaudeCode 的 172 个应用场景（8）：企业培训


前面一篇讲的是项目方案与交付。

这一篇讲第八大类：企业培训。

企业培训这件事，表面看是“做 PPT、上台讲课、现场演示”。但真正做起来会发现，它不是单纯讲课，而是一整套交付流程：要不要做培训，课程怎么分层，培训大纲怎么写，PPT 怎么做，Demo 怎么选，测试数据怎么准备，口播稿怎么写，现场怎么防翻车，讲完以后怎么交付资料，最后还要复盘哪里讲得好、哪里讲得差。

我以前对培训这件事也有犹豫。因为我不是传统讲师型的人，不喜欢站在台上照着 PPT 表演，也不擅长讲那种特别标准化、特别“培训班”的内容。但后来发现，企业 AI 培训不一定非要走讲师路线。我的优势不是会讲课，而是有真实项目经验、知道企业落地会卡在哪里、能把 AI 工具和业务场景结合起来讲。

所以我对培训的理解也慢慢变了：它不是单纯卖课，而是企业 AI 落地的入口。培训可以帮助客户建立认知，也可以暴露客户真实需求，还可以把后续咨询、智能体定制、私教陪跑接起来。

这一篇一共 24 个场景。

## 1. 培训该不该做 / 课程分层的战略讨论

这是做企业培训之前最早遇到的问题：我要不要做培训？如果做，到底做成什么形态？

原来我对培训有点纠结。一方面，企业确实有 AI 培训需求；另一方面，我又不是传统讲师型的人。传统培训往往需要很强的舞台感、控场能力、PPT 表达能力，而我更擅长的是实战、分析和项目落地。

如果只是跟市场上那些讲师拼“谁更会讲”，我不一定有优势。但如果把培训定位成“企业 AI 落地的入口”，它就不是普通讲课，而是把我的项目经验、工具能力和业务判断包装成一种产品。

现在我会让 Claude Code 帮我把这个问题拆开讨论。

输入是：我的个人定位、性格特点、过往项目经验、和培训行业朋友的聊天录音，以及我对培训的顾虑。

输出是：要不要做培训的判断、培训产品怎么分层、每一档解决什么问题、适合什么客户。

比如可以拆成三档：半天课解决认知拉齐和兴趣激发，适合老板、高管或业务负责人；一天课解决工具入门和场景演示，适合企业核心团队；两天课解决实操训练和场景共创，适合某个部门或项目组，最好结合客户业务做真实数据、真实流程和真实 Demo。

这件事省心的地方是：Claude Code 把“我到底要不要做培训”这种模糊的业务纠结，拆成了产品结构和交付路径。

它也帮我明确了一个判断：我没必要把自己包装成传统培训讲师，而应该走“实战派顾问型培训”。先用内容预筛客户，再用专业度建立信任，再看是否转咨询和项目。

## 2. 培训资料收集与培训知识库框架搭建

决定做培训之后，第一件事不是马上做 PPT，而是建素材库。

原来做培训材料很容易东拼西凑：飞书里有一些资料，公众号里有一些文章，播客里有一些观点，短视频里有一些案例，过去项目文档里也有一些内容。每个材料都可能有用，但散在不同地方，很难系统使用。

现在我会让 Claude Code 帮我搭培训知识库框架。

输入是：飞书 wiki、公众号文章、播客摘要、短视频素材、项目文档、架构图截图、客户沟通记录，以及我手上已有的培训资料。

输出是：Obsidian 里的「培训材料」分级文件夹结构，把资料分门别类归档。

比如可以按基础认知、企业场景、工具教程、案例素材、Demo 数据、口播稿、客户行业、培训复盘这些方向拆。后来如果发现结构太复杂，还可以简化成一二级目录，保证能长期维护。

这件事省心的地方是：培训不再是临时搭 PPT，而是先建立一套可复用的素材库。

企业培训最怕空。PPT 页数很多，但里面全是泛泛概念，没有真实案例，没有真实 Demo，没有能让客户回去就用的东西。培训知识库的价值，就是给课程持续提供弹药。

## 3. 培训大纲设计与多轮迭代

接到企业 AI 培训单之后，最先需要对外交付的通常不是 PPT，而是培训大纲。

原来写培训大纲很容易变成一串模块名：AI 认知、Prompt、工具介绍、智能体、案例演示。看起来完整，但不一定贴合客户，也不一定能支撑真实交付。

现在我会让 Claude Code 把客户诉求转成可对外发的大纲，并且跟着客户反馈持续迭代。

输入是：客户背景、对接录音转录、参考别家大纲、实际 PPT、Obsidian 客户沟通记录。

输出是：写进飞书的完整模块化大纲。比如开场演示、认知拉齐、工具地图、工作拆解、Prompt、智能体、Claude Code 入门、Claude Code 应用、Skill、MCP、知识库、综合实操、总结收口。

每个模块还会写清楚：这个模块解决什么问题、讲多长时间、适合放在上午还是下午、Part A/B/C 怎么分、哪些地方需要演示，哪些地方需要练习。

这个场景最有价值的地方，是它能支持多轮迭代。

比如 V1 到 V6：客户希望改模块名，就改模块名；希望删掉某部分，就只删那部分；希望加时长，就补时长；希望分上下午，就重排结构；希望突出某个工具，就调整模块顺序。Claude Code 会严格按“只改指定模块，其他不动”的方式处理，避免一改把整份大纲弄乱。

这件事省心的地方是：一次培训大纲不是一次性文档，而是可以沉淀成可复用模块库。得物、成都建行、宝尊、千织，不同客户的课可以在一套模块体系上调整，而不是每次从零写。

## 4. 培训框架复盘与重构建议

有时候大纲写出来了，但不一定真的打中客户需求。

原来我自己看培训框架，很容易陷在现有结构里。比如已经写了 AI 认知、工具、Prompt、Agent，就会继续在这些模块里修修补补。但客户真正的问题可能不是这些，而是他们到底怎么从聊天工具走到可执行工作流。

现在我会让 Claude Code 跳出来评估整个框架。

输入是：当前模块清单、客户诉求、客户现有 AI 基础、客户最大堵点。

输出是：结构性问题诊断和重构建议。

比如它可能会指出：Claude Code 比重过重，飞书扣子没有独立模块，长期记忆和定时触发没讲透，学员如果只停留在 chatbot 逻辑里，后面讲 Agent 会断。

然后它会抛开原框架重新组织：为什么要学 AI，应该拆什么工作，Agent 怎么拆，怎么真做，怎么落地。也会判断飞书扣子和 Claude Code 应该如何双线分工：哪个适合业务人员，哪个适合更复杂的执行系统。

这件事省心的地方是：它不只是帮我改字，而是帮我判断“这个课到底有没有打中客户”。

培训框架最怕看起来很全，但主线不清。Claude Code 可以帮我从客户视角重新审一遍，必要时推倒重组。

## 5. Claude Code 培训框架评审

有时候我要专门给别人讲 Claude Code。

这类培训不能只是讲“怎么安装、怎么用命令”。真正要讲清楚的是：Claude Code 和普通聊天工具有什么区别，和 Hermes 有什么区别，和豆包这类超级智能体有什么区别，Claude Code 到底由哪些部分构成，为什么它适合做复杂工作流。

原来我会自己搭框架，但容易漏一些关键概念。比如讲了 MCP，没有讲 Skill；讲了知识库，没有讲 CLAUDE.md；讲了工具调用，没有讲为什么它更像工作操作系统。

现在我会让 Claude Code 对我的讲解框架做评审。

输入是：Claude Code 培训大纲，比如 CC 与 Hermes 区别、CC 与豆包区别、CC 由什么构成、CC 怎么用。

输出是：框架评审意见。它会判断这个结构是否完整，哪些地方容易讲混，哪些概念要前置，哪些对比能帮助学员理解。

比如 Claude Code 可以拆成 MCP、Skills、知识库、CLAUDE.md、文件系统、命令行、上下文规则这些组成部分。Hermes 更像某种封装好的 Agent 产品，而 Claude Code 更像可控的工作环境。豆包这类工具更适合普通用户直接用，Claude Code 更适合把工作流工程化。

这件事省心的地方是：培训前先校验讲解逻辑，避免讲成碎片工具介绍。

## 6. 飞书提效培训课程设计

有些客户会提出很具体的培训需求，比如“帮我们讲一下 AI 办公，重点讲飞书如何提效”。

这种需求如果处理不好，很容易变成功能介绍：飞书有文档、会议、审批、多维表格、知识库、智能助手，然后一项项罗列。客户听完可能知道飞书功能很多，但不知道自己该怎么用。

现在我会让 Claude Code 帮我重新设计课程结构。

输入是：客户原始需求，比如“飞书为重点，先整体后实操”，再加上我的个人知识库和已有文章素材。

输出是：一套课程结构。比如整体部分占 30%，讲飞书不是一个单点工具，而是 AI 串联的工作操作系统；实操部分占 70%，围绕会议、文档、检索、审批四个场景展开，用“旧方式 → AI 新方式”的对比来讲。

这个场景的关键是：不要罗列功能，要讲 AI 解决了什么旧问题。

比如会议场景，旧方式是开完会没人整理、行动项容易丢；新方式是自动转写、提炼纪要、生成待办。文档场景，旧方式是人工写、人工改、版本混乱；新方式是 AI 帮助起草、总结、重写、归档。审批场景，旧方式是流程不透明、信息不完整；新方式是 AI 帮助预填、审核、解释制度。

这件事省心的地方是：课程不再围绕产品功能展开，而是围绕客户工作场景展开。

客户不关心飞书有多少按钮，他关心自己每天的会、文档、审批、检索能不能更省事。Claude Code 可以帮我把工具讲成场景，而不是讲成说明书。

## 7. 培训 PPT 整体审阅与结构/逻辑诊断

培训 PPT 做出来之后，不能马上拿去讲。

很多 PPT 看起来页数够了，但逻辑不一定顺。可能某些页太虚，某些页技术味太重，某些案例和受众无关，某些模块时间撑不住，某些概念讲法容易误导。

原来我自己改 PPT，容易陷在局部。看某一页觉得还行，但整体节奏不一定对。特别是几十页、上百页的 PPT，人工从头到尾抠逻辑很累。

现在我会让 Claude Code 做“内容医生”。

输入是：培训 PPT，或者某一页 PPT 加上我的疑问。

输出是：结构合理性判断、哪些模块该调整、哪些案例该删、哪些 Demo 不适合、哪些概念需要重讲、哪些页面逻辑不成立，以及具体改写建议。

这个场景里，Claude Code 不一定直接改 PPT，而是先评估。比如它会读每页内容和备注，结合受众判断这页有没有必要。给财务部门培训，就要判断 Demo 是不是财务岗回去能用；给电商客户培训，就要判断案例是不是贴近电商业务；给高管讲，就要判断是不是太工具化、太技术化。

它也会帮我逐页抠一些概念，比如 AI 分水岭四时刻、GPT 三个字母、智能体定义、Day 回顾页这些地方，逻辑是否顺、表达是否准确、客户能不能听懂。

这件事省心的地方是：一份大 PPT 可以被系统体检。

培训 PPT 不能只看“有没有内容”，还要看“客户听完会不会觉得有用”。Claude Code 可以先把不合理、超时、敏感、技术味太重的页挑出来，避免到现场才发现问题。

## 8. 培训课件 PPT 批量制作 / 模块重做

审完结构之后，还要把课程落成正式 PPT。

原来做一套企业培训 PPT 非常耗时间。两天课可能要 200 到 270 页，一天课、半天课还要另外裁剪。某个模块客户不满意，还要保持原风格重做。手工逐页做，很容易风格不一致、排版溢出、内容太乱。

现在我会让 Claude Code 帮我批量制作和重做 PPT。

输入是：课程框架、模块内容、深度偏好、参考 PPT、已有视觉风格。

输出可以是几类：一套商务风 .pptx、一份逐页清单、某个单页或模块重做版，也可以同时生成 PPT 和横向翻页网页 PPT HTML。

Claude Code 可以先做两天版，再反推一天版和半天版。大型 PPT 可以拆成多个并行块，分别写 slide JS，再用脚本校验风格一致性。风格上保持白底、统一主题色、固定排版规则。某个模块要重做时，它会先去 Obsidian 深研素材，再重述需求确认，最后动手。

这个场景里，返工也很常见：太丑、太乱、字太多、溢出、逻辑不清，都可以继续改。

这件事省心的地方是：把课程内容直接落成可用 PPT，省掉大量逐页手搓的时间。尤其是同一套内容需要多种交付形态时，Claude Code 的批量生成和风格保持能力很有价值。

## 9. 培训现场演示 Demo 设计、选型与合理性评估

企业 AI 培训最重要的不是讲概念，而是现场 Demo。

但 Demo 选不好，很容易翻车。太技术化，客户看不懂；太简单，客户没感觉；太敏感，比如涉及邮件、内部数据，现场可能不方便演示；太复杂，时间不够；和受众岗位无关，客户会觉得“这个和我有什么关系”。

原来我准备 Demo，更多靠经验判断。比如想演示发票识别、制度问答、经营数据分析、合并报表、自动化办公。但这些 Demo 哪些该保留、哪些该换、哪些适合财务、哪些适合管理层，需要逐个判断。

现在我会让 Claude Code 帮我评估 Demo 列表。

输入是：PPT 现有 Demo 列表、客户岗位、课程时长、已有场景库，比如财务 15 大场景和 75 个具体用法。

输出是：每个 Demo 的保留、替换、新增建议；每个 Demo 需要准备什么材料；现场时间是否够；哪些 Demo 有敏感风险；哪些 Demo 的 WOW 效果更强。

比如对于财务培训，技术味太重的 Demo 可以砍掉，换成费用报销审核、异常检测、预算分析、发票识别、制度问答、报表合并这些财务刚需。

这件事省心的地方是：Demo 选型有依据。

培训现场最怕“我觉得这个 Demo 很酷，但客户没感觉”。Claude Code 可以帮我从受众岗位、业务痛点、现场可演示性、时间成本、翻车风险这几个维度提前筛一遍。

## 10. 培训 Demo 配套：测试数据、材料、演示 prompt、前端对话页

Demo 选好了，还要准备材料。

原来做一个 Demo，可能要自己造数据、写制度文档、准备 FAQ、设计演示 prompt、整理文件夹。如果要做多个 Demo，这个准备量很大。

现在我会让 Claude Code 批量生成 Demo 配套材料。

输入可以是很具体的需求，比如“造一份详细的财务制度文档”“数据再复杂一点”“三张表按订单号合并”“做一个客服智能体演示页面”“给这个 Demo 写一两句现场能念的 prompt”。

输出可以是多文件测试数据、制度文档、FAQ、演示 prompt，甚至一个单文件 HTML 对话页。

比如制度问答 Demo，需要一份足够真实的财务制度文档；费用分析 Demo，需要一份有异常、有环比、有部门和科目的数据；合并报表 Demo，需要多张表之间有可对比字段；客服智能体 Demo，如果没有真实系统，也可以用 HTML 做一个输入框，写死样例逻辑，模拟“输入 → 判断场景 → 回复”的交互。

这个场景里，我很看重演示 prompt 的压缩。现场讲 Demo，不能把 prompt 写得很长很复杂，否则客户会觉得 AI 是靠你写了一大段指令才工作的。更好的方式是把演示指令压到两三句，让客户感受到“自然语言驱动”的力量。

这件事省心的地方是：现场实操需要的全套材料，都可以批量准备出来，不用手工拼数据。

## 11. 培训项目流程图 / 工具对比表

培训课件里经常需要图和表。

比如讲一个客服 Agent 项目，光用文字说“以前怎么处理，现在 Agent 怎么处理”，学员不一定有感觉。画成新旧流程对比图，就更清楚。

再比如讲为什么推荐 Claude Code，不能只靠口头说“它更强”。最好有一张对比表，把 Claude Code、Hermes、飞书小龙虾等工具放在一起，看 skills、CLAUDE.md、MCP、插件、知识库这些能力分别支持到什么程度。

原来这些图表都要手工画，特别是 Mermaid 流程图，经常要反复调节点方向、顺序、可渲染性。表格也要考虑飞书 Markdown 能不能正常显示。

现在我会让 Claude Code 直接生成。

输入是：项目流程描述、工具对比维度，或者某个项目的新旧流程。

输出是：可以放进 PPT 的 Mermaid 流程图，或者可以直接贴进飞书 Markdown 的对比表。

比如客服 Agent 的流程图，可以画成旧流程和新流程上下对比：旧流程是人工接单、人工查资料、人工回复；新流程是用户输入、意图识别、知识库检索、Agent 生成回复、人工审核。对比之后，Agent 的价值会更直观。

这件事省心的地方是：课件里的复杂流程可以被压缩成一张图。

图表不是装饰，而是帮助学员理解。尤其是企业培训，很多人听概念会走神，但看到流程对比，就容易明白“AI 到底替代了哪一步”。

## 12. 培训逐字稿 / 口播稿撰写 + 模块时间评估

PPT 做完之后，还有一个很现实的问题：怎么讲？

很多培训不是 PPT 做完就能讲。尤其是两天课、上百页 PPT，如果没有口播稿，现场容易讲散、讲超时、漏掉重点，或者临场组织语言太累。

原来我需要自己对着 PPT 一页页想：这页怎么开口？讲多久？哪些话要强调？哪些地方要引导互动？第一天下午怎么撑满？哪些模块可能时间不够？

现在我会让 Claude Code 根据 PPT 写逐字稿和时间评估。

输入是：完整 PPT 文本、每页讲师备注、写作指南、页码区间和时间分配。

输出是：每页一个口播块，比如 `## P12｜标题`，下面有预计时长，以及第一人称口语稿。它会按 200 字/分钟左右控制字数，并逐页推算更合理的模块时长。

这个场景里还有一个细节：不要只按 PPT 字数估时间。很多页字少但要讲很多背景，很多页字多但可以快速带过。所以 Claude Code 会根据实际讲解内容估时长。

如果 PPT 页码顺序和任务要求不一致，它也会按真实结构调整，并重新分配时间。整套两天课可以按模块切给多个会话并行写，再汇总。

这件事省心的地方是：大体量 PPT 可以变成逐字可念的讲稿。

我不是要照着稿子死念，而是提前把讲法想清楚。讲师不需要现场重新组织语言，时间也更容易控制。

## 13. 培训考核题 / 课程互动问答设计

企业培训现场还有一个问题：学员不一定主动提问。

如果现场没人问，讲师就会很尴尬。培训结束后，如果没有测验，也很难知道学员到底听懂了多少。

原来我需要自己出题。选择题、判断题、问答题，看起来不难，但要做到不超纲、有对错、难度合适、能现场互动，也不容易。

现在我会让 Claude Code 根据培训大纲或当天模块范围出题。

输入是：培训大纲、当天模块内容、题型要求。

输出可以是 10 道以内的选择题、判断题、问答题，带答案和解析；也可以做成每题一页、点击展示答案的交互 PPT；还可以准备一组现场自问自答问题，用来防冷场。

这里有几个约束很重要：题目必须来自课程内容，不能超纲；选择题和判断题必须有明确对错；不能出那种“请调研一下”的开放题；答案不能超过学员当天学过的范围。

这件事省心的地方是：测验题和互动题可以从课程内容自动生成，既能检验学习效果，也能帮讲师控制现场氛围。

## 14. 课程应用场景分类设计

同一套课程，给不同客户讲，应用场景不能完全一样。

给银行对公客户经理讲，和给电商员工讲，重点肯定不一样。前者可能关心客户资料整理、授信材料分析、风险提示、客户沟通；后者可能关心商品图、客服、内容生产、数据分析、运营提效。

原来如果只是拿一套通用课纲去讲，很多案例会离学员太远。学员听完会觉得“这个很好，但和我没关系”。

现在我会让 Claude Code 根据受众画像重新设计应用场景分类。

输入是：受众画像、现有六大类应用场景、客户岗位、课程目标。

输出是：贴合该受众工作的场景划分和可演示案例。

比如给建行对公客户经理讲，就要考虑他们授信经验不深、需要客户资料分析、行业研究、风险提示、沟通材料整理。给宝尊电商员工讲，就要考虑他们可能还停留在 chatbot 逻辑里，需要先从工作拆解、内容生产、客服、运营数据这些场景切进去。

这个场景里有一个关键要求：场景要能写进去、能演示，不能虚。

这件事省心的地方是：同一套课纲可以按客户快速定制。不是每个客户从零做课，而是在通用能力框架上换成他们听得懂、用得上的场景。

## 15. 工作拆解方法论设计 + XMind 拆解图编辑

企业培训里，我很重视“工作拆解”。

因为很多人不会用 AI，不是因为不会写 Prompt，而是因为不知道自己的工作怎么拆。工作没拆清楚，AI 就不知道该帮哪一步。

原来我讲工作拆解，可能只能靠口头解释：先把岗位任务拆成流程，再看哪些环节能交给 AI。但如果没有方法论和图，学员不容易照着做。

现在我会让 Claude Code 帮我把这个模块做成教学样本。

输入可以是两类。

第一类是我自己过去写过的公众号文章。Claude Code 会从里面抓取拆解方法论，提炼成一页 PPT：怎么拆工作，怎么判断 AI 能介入哪一步，怎么把任务转成输入输出。

第二类是我的业务工作流，比如获客、转化、交付。Claude Code 可以读取或创建 XMind，把每个环节拆开，再把 AI 能协助的任务标特殊颜色，最后按拆解结果填评估表。

输出是：方法论页、XMind 拆解图、任务评估表。

这件事省心的地方是：把我的零散经验变成可教、可演示、可练习的方法论。

学员听完不是只知道“AI 很有用”，而是知道怎么把自己的工作拆出来，再判断哪些环节适合用 AI。

## 16. 培训案例真实性验证 / 案例重造

培训里最怕讲假案例。

有些案例听起来很顺，但数据不真实；有些案例和前面讲的原理对应不上；有些案例是网上来的，讲起来没有细节；有些反面案例丢了素材，需要重新找。

原来我只能凭记忆判断案例能不能用。尤其是 PPT 里已有的案例，过一段时间再看，可能忘了来源和真实性。

现在我会让 Claude Code 帮我做案例真实性验证。

输入是：PPT 里的案例、前面讲的原理、历史动态或素材库。

输出是：这个案例是否站得住、和前面原理是否对应、哪些细节需要补充、哪些地方不能讲太满。如果需要反面案例，它还可以从历史即刻动态 CSV 里找我自己的翻车案例，做成更真实的反面素材。

这件事省心的地方是：培训案例更可信。

企业客户其实很容易识别空案例。你讲一个特别漂亮但没有细节的故事，客户未必信。反而是有真实过程、有失败、有复盘的案例，更能建立信任。

## 17. 用知识库素材增强培训 PPT

有时候 PPT 已经做出来了，但某些模块还是薄。

比如得物培训 PPT 有 187 页，但 Skill 模块可能缺好坏判断标准，知识库模块可能缺知识资产框架，Prompt 和 MCP 的关系讲得不够清楚，综合实操需要更多真实案例。

原来我如果想补 PPT，只能自己去知识库里翻文章。素材很多，但很难快速找到和某个模块最匹配的内容。

现在我会让 Claude Code 用 Obsidian 知识库反向增强 PPT。

输入是：培训 PPT、要增强的模块，比如 Skill、知识库、认知拉齐、综合实操。

输出是：每个模块该补什么的具体建议，并且标出处。

比如 Skill 模块可以补“好 Skill 和坏 Skill 的判断标准”“Skill vs Prompt vs MCP 的对比表”；知识库模块可以补“知识资产框架”“企业知识库为什么不是上传文档”；认知拉齐模块可以补更适合财务人的例子。

这个场景的关键是“逐模块补充”。Claude Code 会先提取 PPT 全文和备注，再到 Obsidian 内容创作里按主题搜相关文章，逐篇阅读后和模块内容交叉比对，最后给补充清单。

这件事省心的地方是：散落在知识库里的优质素材，可以系统性灌进 PPT。每个补充建议都有来源，不是临时编。

## 18. 培训素材池建设

企业培训要长期做，就不能每次都临时找素材。

我需要一套自己的素材池：真实项目案例、原创方法论、内容素材、工具对比、行业素材、角色话术金句。

原来这些东西散在 AI AS ME、CLAUDE.md、内容创作、客户沟通、经验库、教训库里。要用的时候才去找，很低效，也容易漏。

现在我会让 Claude Code 帮我系统整理培训素材池。

输入是：搜索目录、模板、约束。比如只要本人真实项目和原创方法论，不混网上案例。

输出可以分成几类。

第一类是已落地项目案例卡片。每个案例包括业务背景、技术架构、效果、可讲细节、适用课程版本和来源。

第二类是原创方法论。比如判断值不值得做、项目方案标准流程、客户分级、四步分析法、爆款三框架、RAG 分层排查法。每条都包括观点、展开、适用场景、举例和来源。

第三类是内容素材索引，比如原创文章、按行业案例、按角色话术金句。

第四类是工具类素材，比如不同大模型实测、Qwen vs Opus、Agent 平台对比等。

这件事省心的地方是：培训需要的弹药可以集中管理。更重要的是，它能暴露缺口。找不到的就写“未找到”，提醒我后续补。

长期来看，这个素材池比单次 PPT 更有价值。因为每次新客户、新课程，都可以从这里取料。

## 19. 培训现场录音转录稿精读复盘

培训讲完之后，真正重要的是复盘。

原来我讲完两天课，可能凭感觉觉得哪里好、哪里不好。比如感觉某些地方讲得太快，某些 Demo 不够顺，某些互动冷场。但这种复盘很主观，也很容易美化自己。

现在我会把现场录音转写稿交给 Claude Code，让它往死里挑。

输入是：两天培训的完整转录稿，可能有 8000 多行、16 小时，还带时间戳；以及我指定的复盘视角。

输出是：带原文引用和时间戳佐证的复盘报告。

比如它会指出 5 个致命问题：时间失控、看懂不等于做出、演示翻车、互动冷场、财务场景映射太晚。也会保留正反馈，比如哪些案例有效、哪些表达引起学员反应、哪些互动值得继续保留。

最后它会整理出“课前 / 内容 / 现场”三段可照做清单。

这个场景里，Claude Code 处理大文件时不能直接读完，就会先用命令查规模，找 Day1、Day2 边界，再把上午、下午分给多个子 Agent 并行提取节奏、反应、Q&A、能量低谷信号，最后主 Agent 汇总。

我会特别要求它：这是复盘，不是总结；不要美化；要带原话；要指出问题。

这件事省心的地方是：培训复盘从凭感觉，变成有证据、有时间戳、有改进动作。

一场培训真正变成能力资产，不是靠讲完，而是靠复盘。尤其是两天课这种重交付，只有把问题拆出来，下一场才会明显变好。

## 20. 培训交付物清单梳理 + 批量生成交付文件

培训结束后，客户通常还需要交付资料。

原来我可能要自己想：该给客户什么？PPT 要不要给？练习材料要不要给？Demo 数据要不要给？提示词要不要整理？制度文档、FAQ、案例、作业模板要不要打包？

如果靠临时整理，很容易漏。

现在我会让 Claude Code 结合培训大纲和实际 PPT，梳理交付物清单，并批量生成文件。

输入是：培训大纲 PDF、实际 PPT、讲师备注和课程内容。

输出是：应该交付哪些文件的清单，以及按内容生成出来的交付文件夹。比如课件、练习材料、Demo 数据、提示词清单、制度问答材料、表格分析样例、课后作业说明等。

它还可以按客户要求平铺到一个文件夹里，不用我一个个手工整理。

这件事省心的地方是：交付物从“我想想该给什么”，变成“按课程内容自动生成成品”。

培训不是讲完就结束。客户拿到资料以后，员工能不能继续练，能不能复盘，能不能带回部门使用，很大程度取决于交付物是否完整。

## 21. 培训课前准备清单

企业培训最怕现场翻车。

尤其是涉及 Claude Code、GitHub、内网、飞书、投影、网络、权限、API、安装环境这些东西，任何一个环节卡住，现场体验都会受影响。

原来课前准备靠记忆，很容易漏。比如 HDMI 转接头没带，酒店网络不稳定，客户电脑没有权限，GitHub 打不开，Claude Code 环境没装好，内网访问不了，投屏比例有问题。

现在我会让 Claude Code 根据培训内容反推课前准备清单。

输入是：培训项目情况、现场实操内容、需要用到的工具和环境。

输出是两份清单：客户需要准备的，我自己需要准备的。

客户需要准备的可能包括：参训人员电脑、网络权限、工具账号、是否能访问 GitHub、是否允许安装软件、是否有内网限制、是否需要提前准备业务数据。

我自己需要准备的可能包括：Claude Code 权限、Demo 环境、备用网络、HDMI 转接头、转换器、激光笔、提前到酒店试投影和网络、离线备份材料。

这件事省心的地方是：提前排雷。

很多培训不是内容不行，而是现场环境把节奏打断。课前清单越细，现场越稳。

## 22. 别人出的培训大纲审核 + 对外说辞 / 交付物质检

有时候一场培训不是我亲自讲，而是转给合作伙伴或外部讲师。

这时候我需要做两件事：第一，审对方大纲和课件质量；第二，准备向客户解释为什么是合作讲师来讲，而不是我本人讲。

原来我只能自己看大纲，凭感觉判断行不行。然后再临时想话术，告诉客户“这是我的合作讲师”。如果讲不好，还是会影响我的信誉。

现在我会把对方的大纲或课件交给 Claude Code 质检。

输入是：对方提供的培训大纲 PDF，比如财务体系 AI 培训大纲，以及客户背景。

输出是：大纲问题点评、内容是否合理、是否贴近客户、案例是否足够、有没有太泛、有没有缺实操，以及一段对外介绍合作讲师的说辞。

这段说辞的重点是做背书包装。比如强调这是我长期合作的讲师，有相关行业经验，有过财务案例或企业交付经验，这场课由他主讲，我负责整体方案和交付质量。

这件事省心的地方是：外部交付物可以快速质检，对外沟通也有统一口径。

如果我的业务要规模化，后面一定会遇到合作讲师和交付伙伴。不能每次都完全靠信任，必须有内容审核和对外包装机制。

## 23. 培训客户沟通框架与回复话术

最后一个场景，是培训客户咨询时的沟通框架。

有人来问培训，不一定马上知道要怎么聊。有些客户只是问问，有些客户有明确预算，有些客户只想要大纲，有些客户想看公司介绍，有些客户其实需要的是定制课。

原来我可能会凭感觉回复：先介绍一下我们能讲什么，再问对方需求。但如果没有框架，很容易问得太散，或者过早发方案。

现在我会让 Claude Code 帮我准备培训咨询的回复和沟通框架。

输入是：客户背景，比如宁波外贸公司，或者客户发来的咨询消息。

输出是：该向客户了解哪些信息、应该怎么回复、是否该先发大纲或公司介绍、是否需要约电话、怎么突出我的实战经验差异化。

比如要先问清：培训对象是谁，是老板、高管、全员还是某个部门；培训目标是什么，是认知、工具、场景共创还是项目转化；时长预期是多少；是否有行业场景；是否需要现场实操；客户更关注通识还是落地。

这件事省心的地方是：每次培训咨询都有一个标准开场框架。

同时，它也会帮我区分客户。如果是懂行、高客单价、需要定制的客户，我可以亲自做；如果是不懂 AI、预算不高、需求偏通识，可以交给合作讲师。这样不会所有线索都用同样精力处理。

## 24. 培训定价策略测算

培训最后还有一个很现实的问题：到底报多少钱。

原来我容易按“每天多少钱”来报。比如过去报过 6000 元/天，但如果算上准备 PPT、往返、讲课、课后交付，真实成本可能是 4 到 5 天。这样一算，6000 元/天其实并不高，甚至可能偏低。

现在我会让 Claude Code 帮我做定价策略测算。

输入是：历史报价、真实准备成本、交付内容、客户类型、课程时长。

输出是：报价判断和理由。比如应该提到 1 万/天，或者不要按天卖，而是按项目包、成果包来卖。

它还会帮我重构报价单：把培训从“几天课”改成“带交付物的实战课”。报价里不只是讲课时间，还包括前期调研、课程定制、PPT、Demo 数据、实操材料、课后交付物、答疑和复盘。

这件事省心的地方是：用成本测算支撑涨价决策，避免低价透支时间。

培训如果只按“讲一天”卖，很容易被客户拿来和普通讲师比价。但如果按“成果包”卖，客户看到的是一整套交付，而不是一个人站上去讲几个小时。

## 这类场景真正解决了什么问题

企业培训这一类场景，本质上解决的是“培训产品化”和“培训交付标准化”的问题。

如果只是临时做一场课，可能准备一份 PPT、讲一讲就结束了。但如果想长期做企业 AI 培训，就不能只靠临场发挥。

要先判断培训是不是适合自己。  
要把课程拆成不同层级。  
要有培训素材库。  
要能为不同客户设计大纲。  
要能审 PPT、做 PPT、改 PPT。  
要能选 Demo、造数据、写 prompt。  
要能写讲稿、估时间、设计互动题。  
要能做课前准备、课后交付、最后复盘。  
还要能给培训定价，判断这场课值多少钱。

Claude Code 在这里的价值，不是替我上台讲课，而是把培训背后的大量准备工作、交付工作和复盘工作系统化。

它可以帮我做课程设计。  
可以帮我搭素材库。  
可以帮我审 PPT。  
可以帮我做 PPT。  
可以帮我选 Demo。  
可以帮我造测试数据。  
可以帮我写口播稿。  
可以帮我设计互动题。  
可以帮我做课前 checklist。  
可以帮我整理交付物。  
可以帮我从 16 小时转录稿里复盘问题。  
也可以帮我测算价格，避免把自己的时间卖便宜。

这里有几个关键原则。

第一，企业培训不能只讲概念。客户真正有感觉的，是能不能看到和自己岗位相关的场景、案例和 Demo。

第二，课程大纲要能迭代。不同客户、不同受众、不同 AI 基础，课纲不能完全一样。最好有模块库，再按客户重组。

第三，Demo 选型比 Demo 数量更重要。一个贴近客户岗位的 Demo，比十个炫技 Demo 更有价值。

第四，课前准备决定现场稳定性。工具、网络、权限、投影、数据、备用方案，都要提前检查。

第五，培训交付不是讲完就结束。交付资料、练习材料、课后作业、复盘报告，都是客户感知价值的一部分。

第六，复盘要基于原始记录。不要只凭感觉说讲得好不好，最好从录音转写里找证据，带时间戳地复盘。

第七，培训要按价值报价。不要只卖“几小时讲课”，要把调研、定制、Demo、材料、交付物和复盘都算进成果包里。

## 下一篇写什么

这篇讲的是第八大类：企业培训。

下一篇可以继续拆第九大类：调研与情报。

如果说企业培训解决的是“怎么把 AI 讲给客户听并交付一场课”，那调研与情报解决的就是另一类问题：怎么快速研究一个行业、一个公司、一个产品、一个工具、一个政策或一个竞争对手，并把它变成可用于销售、方案、内容和决策的判断。
