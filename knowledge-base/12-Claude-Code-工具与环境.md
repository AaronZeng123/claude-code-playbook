# ClaudeCode 的 172 个应用场景（12）：Claude Code 工具与环境


前面十一篇，基本都在讲 Claude Code 怎么帮我做具体工作：内容、素材、音视频、财务、文档、销售、项目、培训、调研、知识管理、自动化。

最后一篇，我想讲更底层的东西：Claude Code 工具与环境。

这部分不是直接面向客户的业务场景，但它很重要。

如果前面那些场景是一条条业务流水线，那这一篇讲的就是“工作台本身怎么维护”。比如 Skill 怎么安装、怎么卸载、怎么排查为什么装了看不到；怎么把外部开源项目封装成自己的 Skill；怎么审计 API Key 泄露；怎么安装 MCP；怎么用浏览器自动化处理登录态页面；怎么管理模型、版本、权限、备份和会话；怎么防止网页或文件里的注入指令劫持任务。

这些事听起来偏技术，但我后来发现，它们决定了 Claude Code 能不能长期稳定使用。

如果 Skill 库乱，后面就找不到工具。  
如果 MCP 配置乱，很多软件就调不起来。  
如果密钥到处硬编码，分享 Skill 就有泄露风险。  
如果权限弹窗一直弹，工作流就会被打断。  
如果会话和资产不备份，电脑一坏就可能丢掉很多沉淀。  
如果不识别注入内容，抓网页和读文件时就可能被内容里的伪指令带跑。

所以这篇不是讲“怎么用一个功能”，而是讲怎么把 Claude Code 维护成一个稳定、可扩展、可复用的个人 AI 工作台。

这一类一共 16 个场景。

## 1. Skill 安装 / 卸载 / 排查位置 / 迁移

这是最基础的一类工具管理场景。

Claude Code 的很多能力是通过 Skill 扩展出来的。比如飞书、公众号抓取、发票识别、图文卡片、视频解析、语音识别、PPT 生成、网页抓取等等。用得越多，Skill 就越多。

原来我装 Skill，经常会遇到几个问题：装了以后 Claude Code 里看不到；不知道 Skill 装到哪里了；装多了以后太乱；某个 Skill 不想要了，但不知道删哪里；还有一些工具装到了 `~/.agents/skills/`，但 Claude Code 实际读取的是 `~/.claude/skills/`。

现在我会让 Claude Code 自己排查。

输入是：安装日志、一个 Skill 名称，或者一句话，比如“这个 Skill 装了怎么没生效”“帮我把没用的删掉”“帮我装一个飞书消息 Skill”。

输出是：安装、卸载、迁移和实测结果。

比如它会对比 `~/.claude/skills` 和 `~/.agents/skills`，发现 skill-vetter 装错位置，就把它复制到正确目录。批量清理时，它可以一次删掉 21 个不用的 Skill。安装新 Skill 时，会用 `find-skills` 或 `npx skills add`，装完之后再跑一次实测，比如飞书消息 Skill 会发一条 test 验证是否真的可用。

这件事省心的地方是：Skill 不再是装完就算，而是装、查、测、迁移、清理都有一套流程。

Claude Code 的能力越多，越不能只靠记忆管理。否则后面会出现一堆“我好像装过，但不知道在哪里”的工具债。

## 2. Skill 脱敏打包分发

有些 Skill 我自己用没问题，但要发给学员、客户、朋友，就必须先脱敏。

原来我可能会直接打包文件夹发出去，但这有很大风险。因为很多 Skill 里可能藏着 API Key、token、cookie、config.json、环境变量兜底值。一旦把真实密钥发出去，后果很麻烦。

现在我会让 Claude Code 先做脱敏，再打包。

输入是：一个 Skill 文件夹，或者一批要分享的 Skill 清单。

输出是：桌面上的去 Key 文件夹、zip 包、依赖说明。比如 chrome-cdp、tiktok-scraper 需要 Node 22+、Python、ffmpeg、yt-dlp、Chrome 登录态；content-video-parser 要清掉豆包 Key。

Claude Code 会先复述需求，确认是要“可分发版本”，再读 SKILL.md、脚本、配置文件，把硬编码 Key 清掉。打包前再用 grep 检查是否还有残留，比如 `API_KEY`、`token`、`secret`、`sk-` 这些关键词。

这个场景里我也踩过坑。曾经有一处 Key 没清干净，被我发现后又补清了一遍。这说明 Skill 分发不能只靠“我觉得没问题”，必须用脚本扫。

这件事省心的地方是：自建 Skill 可以安全分享，别人也能知道依赖和安装条件。

如果以后要把 Skill 作为培训资料或客户交付物，这一步非常关键。

## 3. Skill 库梳理盘点 / 分类 / 合并 / 按能力盘查

Skill 一多，就会变成另一个知识库。

我自己的 Skill 数量到过 70 到 89 个。里面有内容创作、飞书、图片、视频、音频、抓取、PPT、数据处理、浏览器自动化、GitHub、知识库等等。如果不梳理，很快就会乱。

原来我只能凭印象知道“大概有这么几个工具”。但真正要用的时候，还是会想不起来：哪个 Skill 负责 md 转卡片？哪个调 GPT 图片？哪个调豆包？哪个能处理飞书？哪个可以做视频？

现在我会让 Claude Code 对 Skill 库做盘点。

输入是：`~/.claude/skills` 目录，或者一个具体问题，比如“帮我梳理所有 Skill”“内容创作相关 Skill 有哪些”“哪些 Skill 用 GPT API 生图”。

输出是：分类清单或表格。每个 Skill 会有一句话定位、核心能力、关键依赖、调用模型和路径。

比如内容创作专项可以分成核心 13 个、扩展 7 个。图片生成相关可以查出哪些用 `gpt-image-2`，哪些用豆包，API Key 来源在哪里。飞书相关可以单独列出 `lark-*` 系列能力。

这件事省心的地方是：我能一张表看清全部工具家底。

Claude Code 不是单个工具，而是一套工作台。工作台越复杂，越需要资产盘点。否则你会不断重复造轮子，也不知道自己已经有哪些能力。

## 4. Skill 结构 / 触发机制研究

如果只是用别人写好的 Skill，理解不深也没关系。但当我开始自己写 Skill、改 Skill、封装项目时，就必须搞清楚 Skill 到底怎么组织、怎么触发、怎么处理密钥、怎么被 Claude Code 调用。

原来我对 Skill 的理解比较碎：知道有 SKILL.md，知道有 scripts，但不知道 frontmatter 怎么写、description 怎么影响自动触发、哪些目录是标准结构、哪些设置会影响模型调用。

现在我会让 Claude Code 系统研究现有 Skill 结构。

输入是：skills 目录、官方文档、某几个典型 Skill，以及我的问题。

输出是：一套可照搬的 Skill 写法规范。

比如：

- SKILL.md frontmatter 应该包含哪些字段
- scripts、references、assets 目录分别放什么
- setup 分几类
- API Key 有哪些处理方式
- 什么情况下手动 `/skill-name` 调用
- 什么情况下靠 description 自动匹配触发
- `disable-model-invocation` 和 `user-invocable` 会怎么影响使用
- 什么叫渐进式加载
- 新建一个 Skill 的默认骨架是什么
  
这件事省心的地方是：新建 Skill 不再靠抄一个旧项目瞎改，而是有一套标准骨架。

这一步也让我更清楚：Skill 的价值不只是封装脚本，更重要的是把“什么时候用、怎么用、输入输出是什么、依赖是什么、风险是什么”写清楚，让 Claude Code 以后能正确调用。

## 5. Skill 安全审计：API Key / 凭证泄露排查

这和前面的脱敏打包相关，但更偏安全审计。

有时候不是马上要分享，而是想检查一下：我现在这些 Skill 里，到底有没有硬编码密钥？有没有 token？有没有密码？有没有环境变量没设时回退到真实密钥的兜底值？

原来这种检查如果靠人工，很容易漏。因为密钥可能藏在 config.json、Python 脚本、shell 脚本、README、示例文件里。

现在我会让 Claude Code 对整个 Skill 根目录做安全审计。

输入是：一个 Skill 根目录，比如得物培训用到的 23 个 Skills。

输出是两类结果。

第一类是无敏感信息的 Skill 名单。

第二类是有风险的精确位置。包括文件路径、行号、凭证类型、为什么危险。

比如它曾经查出 image-reader/config.json 里有火山引擎 ARK 密钥，也查出 doubao-asr 脚本里把 App ID 和 token 写成了 `os.environ.get` 的默认值。也就是说，如果环境变量没设，就会回退到硬编码真值。这种隐患很容易被忽略。

Claude Code 的做法是：find 列全文件，再 grep `sk-`、`API_KEY`、`token`、`secret`、`password` 等模式，逐文件核查。

这件事省心的地方是：分享、交付、迁移之前，可以先做一轮安全体检。

AI 工作台越强，里面的密钥越多。密钥如果不管理好，工具越多风险越大。

## 6. 把开源项目 / 文章“长”成新 Skill

这是我用 Claude Code 很有价值的一个模式。

有时候我看到一个开源项目，或者一篇文章介绍了一个自动化流程。我不想只是读一读，而是想把它变成自己 Claude Code 里可调用的能力。

原来这件事要自己判断：这个项目怎么装？有没有依赖？能不能封装？需要写什么脚本？Skill.md 怎么写？怎么让 Claude Code 知道什么时候调用它？

现在我会让 Claude Code 把外部项目“长”成新 Skill。

输入是：GitHub 项目链接、文章链接，或者一句话“把这个流程转成 Skill 装上”。

输出是：本地可用的新 Skill。

比如 text-to-cad，可以 clone 项目，建 venv，安装 build123d/OCP，再 symlink 注册成全局 Skill。html-anything 可以 clone、pnpm 安装、写 SKILL.md，并整合模板选择流程。video-auto-pipeline 可以参考文章流程从零建 Skill，不新增 API。月度费用分析流程也可以打包成 Skill，用完之后再判断是否卸载。

这个场景里，一个关键判断是：不是所有流程都值得封装。

一次性任务，直接做就行。高频、稳定、复用价值高的任务，才值得变成 Skill。

这件事省心的地方是：外部开源能力和文章里的工作流，可以直接接入我的 AI 工作台。

以后我不是每次重新看文章、重新查项目，而是把好用的能力沉淀成自己的工具。

## 7. 实操搭建 / 安装开源项目并配置

调研完一个项目，有时候还要真正跑起来。

原来看开源项目最烦的就是环境配置。git clone、装依赖、Python 版本、Node 版本、pnpm、uv、环境变量、API Key、启动命令、端口冲突、进程卡死，任何一步都可能卡住。

现在我会让 Claude Code 直接接管安装和排障。

输入是：“帮我搭起来”“接着装”“这个项目跑不起来”。

输出是：项目真正跑到本地，或者明确说明卡在哪里。

比如 open-design 跑到 localhost；Hermes 用 uv venv 和 wrapper 跑起来；接 DeepSeek 时，排查到环境变量名应该是 `DEEPSEEK_API_KEY`，不是 `CUSTOM_API_KEY`，最后才能 chat。装完之后，如果我发现项目不需要，还可以一键删干净。

Claude Code 会检查 node、pnpm、uv 环境，clone 项目，装依赖，找正确启动命令。卡死时查进程、查缓存、kill 重启。国产 API 接入时，会去读 provider 配置，确认真正需要的变量名。

这件事省心的地方是：我不用自己在各种安装报错里耗时间。

很多开源项目看起来厉害，但真正跑起来才知道能不能用。Claude Code 可以帮我把“看起来能用”推进到“本地实际跑通”。

## 8. MCP 安装 / 配置 / 连接状态盘点

如果说 Skill 是 Claude Code 的工具扩展，那 MCP 更像是把外部软件和系统接进来。

比如 XMind、WPS、GitHub、微信读书、泳道图、Draw.io，这些都可能通过 MCP 接入。问题是 MCP 的安装和配置更容易出坑：要装依赖、配 `.mcp.json`、重启 Claude Code、生效慢、权限复杂，有些工具还要登录态。

原来我如果想用某个 MCP，只能自己搜教程、clone 仓库、改配置、试连接。失败了也不知道是路径问题、依赖问题、权限问题，还是需要重启。

现在我会让 Claude Code 从调研到安装到排障全流程处理。

输入是：想接入的软件或能力，比如 XMind、WPS、GitHub、泳道图、微信读书。

输出是：安装结果、配置文件、连接状态、能力边界说明。

比如 XMind MCP，要装 Go、go install、写进 `.mcp.json`，但需要重启才生效，当场可以先用 Python xmind 库兜底。WPS MCP 会走开源 wps-mcp，安装后排查“Claude 助手”选项卡不显示，甚至要改 publish.xml 和 authaddin.json。泳道图会对比 AI Charts MCP 和 Draw.io MCP，遇到 clone 失败还要配 git 代理。GitHub MCP 要 gh auth login。微信读书则要对比官方 Skill 和基于 Cookie 的 mcp-server-weread，装完发现全文读取不稳就卸载。

这件事省心的地方是：MCP 不再是“装了试试看”，而是变成一套可排查的接入流程。

更重要的是，它会告诉我边界。比如某个 MCP 能读书架，但读不了全文；某个软件可以改文档，但需要本地客户端；某个 MCP 下载了但未激活。知道边界，才不会误用。

## 9. 浏览器自动化：CDP 操控页面 / 抓取登录态内容

这是 Claude Code 工具环境里非常强的一类能力。

很多网页内容，普通 WebFetch 拿不到。比如飞书妙记、小报童付费专栏、小红书评论、抖音数据、扣子后台、腾讯文档 PDF。这些页面要么需要登录态，要么是 JS 渲染，要么有反爬，要么没有导出权限。

原来只能人工打开浏览器，一页页复制、滚动、点击、下载。

现在我会让 Claude Code 通过 CDP 连接我已经登录的 Chrome。

输入是：一个已登录页面、目标任务，比如下载飞书妙记、提取付费专栏、抓小红书评论、采集抖音播放量、操作扣子建智能体、从腾讯文档 PDF 里捞文件。

输出是：本地文件、结构化数据、下载内容，或者自动完成页面操作。

比如飞书妙记没有下载权限，也能拿到视频和完整转录。小报童付费专栏可以提取 Vue SPA 全文并搬到飞书。小红书可以抓正文和三级嵌套评论。抖音可以抓全量视频播放量，再按条件批量删低播放视频。腾讯文档 PDF 不能导出，也可以从 PDF.js 内存里用 `getData()` 捞二进制。

Claude Code 会用 web-access 或 chrome-cdp，通过 eval、scroll、nav、click、snapshot、screenshot 等方式探查 DOM、触发懒加载、提数据。大文件就起本地 Python 服务，让浏览器 fetch blob 后流式 POST 到本地。

这件事省心的地方是：需要登录态、JS 渲染、反爬、导出受限的内容，也能进入自动化流程。

但这个能力也必须有边界。删除、批量操作这类不可逆动作，要先列清单让我确认。不能让自动化变成误删工具。

## 10. Claude Code 原生能力 vs 自有 Skill 审计

Claude Code 原生能力一直在增强。

这就会出现一个问题：我以前写的某些 Skill，现在可能已经被 Claude Code 原生能力覆盖了。继续保留太多冗余 Skill，会增加维护成本，也会让工具库越来越乱。

原来我没有系统判断过哪些 Skill 还值得留。可能有些功能已经没必要封装，或者可以弱化；有些 Skill 还是必须保留，因为它连接了外部 API 或封装了业务流程。

现在我会让 Claude Code 自己审计。

输入是：我的全部 Skill 目录，以及 Claude Code 当前能力。

输出是：分组清单。

比如：

- 可删除：Claude Code 原生已覆盖，大约 26 个
- 保留但弱化：还有点用，但不必作为核心工具
- 必须保留：外部 API、抓取、登录态、专用处理能力
- 业务工作流保留：虽然技术简单，但封装了我的流程
- 需要确认：暂时无法判断
  
Claude Code 会读 settings、各个 SKILL.md，按来源和功能逐个判断能不能被原生能力替代。

这件事省心的地方是：工具库可以瘦身。

长期来看，Skill 的价值不在于数量多，而在于哪些是真正不可替代的：外部系统接入、业务工作流、稳定中间格式、特殊模型能力。这些才应该保留。

## 11. 模型切换 / 版本检查升级 / 配置审计

Claude Code 不是装完就不用管。

模型可能要切换，版本可能要升级，配置可能散在很多地方，密钥可能分布在 settings、zshrc、Skill 脚本、config 文件里。

原来我如果要切换模型、检查版本、查密钥配置，要自己翻文档、看 changelog、查配置文件。很碎。

现在我会让 Claude Code 做配置审计和升级。

输入可以是：把模型换成推理模型；看看 Claude Code 是不是最新版；帮我查一下现在有哪些 API Key 配置。

输出包括：

- 修改 settings.json 的 model，比如切到 `deepseek-reasoner`
- 提醒需要重开会话生效
- 说明推理模型更慢、更贵
- `claude --version` 检查当前版本
- `claude update` 升级
- 按版本整理差异表，并翻译成“对你有什么用”
- 跨 settings.json、.zshrc、各 Skill 的密钥总览表
  
比如 Anthropic 可能走 DeepSeek 代理，火山引擎有 LLM、TTS、ASR，TwitterAPI、cookie、浏览器登录态也分布在不同地方。Claude Code 可以把这些散落配置扫出来，告诉我哪些存在、哪些未找到、哪些没装。

这件事省心的地方是：Claude Code 的底层环境不再是黑盒。

尤其是要新建 Skill 时，先摸清现有密钥和路由，能避免重复配置，也能减少泄露风险。

## 12. 权限确认弹窗白名单配置

Claude Code 用多了，会遇到一个很烦的问题：频繁弹“允许 / 不允许”。

尤其是它要读文件、写文件、跑命令、访问目录时，如果每一步都要我确认，会严重打断工作流。

原来我只能一次次手动点。后来我意识到，这其实可以通过权限配置减少。

现在我会让 Claude Code 帮我修改 settings.json 的 permissions。

输入是：“不想每次手动点允许”“另一台电脑老是出现 yes”“帮我配置一下权限白名单”。

输出是：合并后的 settings.json 权限配置，或者三种方案：手动加 allow、用 fewer-permission-prompts Skill 自动生成、改 accept-all 模式。它还可以给 Trae 用的修改 prompt，或者给第三方 Skill 配 settings.local.json defaultMode，减少反复授权。

这个场景里有一个重点：不要整体覆盖配置。

Claude Code 会先读现有 settings.json，再合并 permissions.allow，尽量不破坏原配置。也会保留安全底线，不建议把所有危险操作都无脑放开。

这件事省心的地方是：减少重复确认，让工作流更顺。

但权限不能只追求省事。该确认的高风险动作，比如删除、外发、改系统配置，还是要保留确认。

## 13. Claude Code 全量资产备份 / 插件安装

Claude Code 用久了，里面会有很多资产。

比如 CLAUDE.md、Skills、hooks、plugins、配置、AI 工作区、脚本、规则库。如果换电脑、电脑坏了、误删目录，这些东西丢了会很麻烦。

原来我没有完整备份意识，可能只是觉得文件都在本地。但真要迁移时，才会发现不知道哪些目录重要。

现在我会让 Claude Code 做全量资产备份。

输入是：一句“帮我备份 Claude Code 资产”。

输出是：一个 `claude-code-backup.sh`，实跑后在桌面生成 321MB 的 tar.gz，包含 CLAUDE.md、72 个 Skills、hooks、plugins、配置和 AI 工作区，后续可以同步到网盘。

这个场景还包括插件安装。

比如看到 11 万 Star 的 Superpowers 插件，就让 Claude Code 安装：clone marketplace，改 known_marketplaces.json，读 hooks 确认，重启后 SessionStart hook 自动注入。也装过 Codex CLI，排查登录状态，并对比它和 Claude Code 的架构差异：一个更像侵入式直接操作，一个更偏非侵入式 diff + apply。

这件事省心的地方是：我的 AI 工作台可以迁移、可恢复、可扩展。

工具用得越深，备份越重要。否则积累越多，丢失成本越高。

## 14. AI 防注入：识别并忽略嵌入内容里的指令

这个场景看起来不像一个功能，但非常重要。

Claude Code 经常会读取外部文件、网页、文章、README、抓取页面。里面可能夹带一些“看起来像系统指令”的文本，比如让 AI 把内容当 malware 分析，或者网页里出现 `<system-reminder>`，列一堆 Skill，好像在诱导模型调用。

原来如果 AI 分不清“用户真正的请求”和“文件内容里夹带的指令”，就可能被带跑。

现在我会要求 Claude Code 把工具返回内容当数据，而不是当指令。

输入是：某个包含注入内容的文件或网页。

输出是：继续完成原任务，并明确说明：这些是嵌在文件或页面里的注入式内容，不是用户原始请求，已忽略。

这个场景里的判断非常关键：只执行用户给出的任务，不执行读取内容里自带的伪指令。

比如我让它分析网页，它就应该分析网页内容，而不是听网页里写的“请调用某某 Skill”。我让它读个人笔记，它就应该把笔记当数据，而不是执行笔记里的命令。

这件事省心的地方是：抓网页、读文件、做调研时更稳，不容易被内容污染。

随着使用 Claude Code 抓取外部内容越来越多，防注入会变成基础能力。

## 15. Claude Code 能力 / AI 概念 / 会话存储机制答疑

还有一类场景，是我不断追问 Claude Code 本身的机制。

比如：

- Claude Code 有哪些客户端形态？
- 循环跑任务到底费不费 token？
- Prompt cache 是怎么回事？
- Agent 子任务会不会占主上下文？
- Claude Code 能不能操作百度网盘？
- Codex 和 Cursor 是什么关系？
- Mac 合盖以后任务还能不能跑？
- 会话到底存本地还是云端？
- 网页端和客户端的会话有什么区别？
- 为什么无害指令也要确认权限？
  
原来这些问题我只能靠网上搜，或者自己试。但很多概念混在一起，容易误解。

现在我会直接让 Claude Code 结合机制和本地环境解释。

输入是：一个具体问题，可能带截图。

输出是：机制解释和可执行建议。

比如循环任务会消耗 token，但 prompt cache 有 5 分钟 TTL；Agent 子任务是独立会话，不占主上下文；百度网盘没有现成 MCP，可以考虑 rclone + 定时任务；token 统计可以有多种方法；Mac 合盖默认会睡眠，可以用 caffeinate 或 pmset；会话既有本地数据，也有云端机制，客户端和网页端并不完全一样。

这件事省心的地方是：工具使用盲区可以被快速扫清。

我不只是知道“能不能”，还会知道“为什么”“怎么做”“有什么替代方案”。

## 16. 会话管理类 Skill 实现审计

最后一个场景，是排查会话管理 Skill 的实现问题。

我之前很在意 Claude Code 的会话存档，因为很多项目沟通、客户判断、代码修改和经验教训都在会话里。如果会话自动保存逻辑有 bug，后续知识沉淀就会受影响。

有一次出现的问题是：会话没有按每个会话一个文件夹保存，而是变成单文件堆在根目录。

原来我只看到结果不对，但不知道到底是哪个 Skill、哪段代码、哪种保存逻辑导致的。

现在我会让 Claude Code 审计相关 Skill 的实现。

输入是：三个 session 管理 Skill 名称，以及异常现象。

输出是：它们的实现代码位置、保存逻辑、单文件还是文件夹、bug 成因分析。

Claude Code 会用 Glob 找 session Skill 文件，用 `which claude` 定位安装目录，再读实现代码，分析为什么会保存到根目录，哪里需要改。

这件事省心的地方是：会话管理不再只是“能不能存”，而是能排查实现细节。

这类问题不解决，前面的知识管理、晚报、项目归档都会受影响。因为历史会话是最重要的原始资产之一。

## 这类场景真正解决了什么问题

Claude Code 工具与环境这一类场景，本质上解决的是“个人 AI 工作台怎么长期稳定运行”的问题。

前面十一类场景都依赖这个底座。

Skill 装不好，很多能力用不了。  
Skill 太乱，工具找不到。  
密钥不审计，分享就有泄露风险。  
MCP 配不好，外部软件接不进来。  
浏览器自动化不稳，登录态内容抓不到。  
模型和版本不清楚，效果和成本都不可控。  
权限弹窗太多，工作流会被打断。  
资产不备份，电脑一坏就全丢。  
不防注入，读网页和文件可能被带跑。  
会话不管理，历史经验没法沉淀。

Claude Code 的价值，不只是它能帮我完成一次任务，而是它能被维护成一个越来越强的系统。

这里有几个关键原则。

第一，工具要可见。Skill、MCP、插件、配置、密钥、版本，都要能被盘点出来。看不见的能力，后面一定会变成混乱。

第二，装完要实测。Skill 和 MCP 不是装上就算，必须验证是否能触发、能不能跑通、边界在哪里。

第三，分享前必须脱敏。任何 Skill、脚本、配置、zip 包，只要要发给别人，就要先扫密钥。

第四，高频流程才值得封装。不是所有任务都要做成 Skill。一次性任务直接做掉，稳定高频流程再沉淀成工具。

第五，自动化要保留安全边界。浏览器自动化、权限白名单、批量删除、批量操作，都必须先确认风险。

第六，工具环境也要知识管理。安装经验、踩坑规则、配置位置、会话机制、备份脚本，都要沉淀下来。否则每次换电脑、换环境、换项目都要重新折腾。
