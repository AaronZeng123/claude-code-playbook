# ClaudeCode 的 172 个应用场景（10）：知识管理


前面一篇讲的是调研与情报。

调研解决的是外部信息怎么查清楚，知识管理解决的是另一个问题：这些信息、经验、案例、规则、素材，怎么真正沉淀下来，并在下一次写作、销售、交付、培训时被重新调用。

这件事对我来说很重要。

因为我用 Claude Code 半年之后，越来越明确一件事：如果没有知识库，AI 就只能帮你处理眼前这一次任务；有了知识库，AI 才有可能越来越懂你、越来越像你的助理，甚至慢慢变成你的工作系统。

知识管理不是简单地“把文件存起来”。它至少包括几件事：把个人背景灌进上下文，把外部长文压缩成可用观点，把自己的 AI AS ME 体系搭起来，把散在各处的素材检索出来，把飞书知识库迁移到本地，把客户案例和方法论资产化，把踩过的坑写成规则，把过去的工作重新整理成叙事。

这一篇讲第十大类：知识管理。

这一类一共 15 个场景。

## 1. 个人/业务知识库 + 外部素材批量灌入做上下文预热

这是最早、也最朴素的一种知识管理方式。

原来我和 AI 对话，每次都要重新交代一遍：我是谁，我在做什么业务，我有哪些客户，我的判断标准是什么，我不喜欢什么表达，我现在有哪些项目。说多了很烦，说少了 AI 又不了解背景。

后来我开始把整套个人知识库批量灌进去，让 AI 先加载上下文。

输入是：人脉资源分级表、公司信息、业务定位、客户获客方法、方法论、异步备忘、外部阅读素材、王川长文、即刻动态、读书笔记等内容。有些会话里，一次粘进去的内容能到几千甚至上万行。

输出很简单：有时候只是让 AI 回复 `ok`，确认它已经读取；有时候 AI 会主动把长文压成结构化摘要，并反问我接下来要做什么、是否需要归档到某个目录。

这个场景看起来很粗糙，但很有用。

它解决的是“新会话冷启动”的问题。AI 一旦读完这些背景，后续整个会话里都会更懂我的业务、我的表达方式、我的资源结构和我的判断逻辑。

比如人脉资源表进去以后，我问某个需求该找谁，AI 就不再泛泛建议“你可以找行业专家”，而是能直接从我的资源里判断谁可能合适。业务定位进去以后，我再讨论客户线索、培训报价、Agent 定制，它也能按我的实际业务，而不是按一个普通咨询公司的模板来回答。

这件事省心的地方是：我不用每次从零解释自己。

它也是后来 AI AS ME 的雏形。先是手工灌上下文，后来才逐步变成更稳定的个人知识库和规则体系。

## 2. 外部长文/读书笔记核心观点提炼

知识管理不只是存自己的东西，也包括处理外部高质量内容。

比如一些长文、读书笔记、投资思考、方法论文章，本身很有价值，但太长。如果只是收藏，最后大概率不会再看。如果只是让 AI 总结，又容易变成普通摘要。

我现在会让 Claude Code 做两层处理。

输入是：外部长文原文，比如王川 2015 年关于错过苹果、油价、寿命预测这类文章，或者一些读书笔记、外部观点材料。

输出不是简单摘要，而是按三层结构压缩：一句话主题、逻辑链拆解、可迁移认知。

如果这篇文章和我当前业务有关，我还会让它继续交叉分析：这篇文对我意味着什么？我该改什么？它和我已有的“判断真不真、值不值、怎么做”方法论有什么关系？

比如有些文章会提醒我，不要因为见过太多伪落地，就对真正的机会过度警惕。这个结论如果只作为读书笔记就很普通，但如果放到我的业务处境里，就会变成一种行为校准。

这件事省心的地方是：读一篇好文，不只是得到几条摘要，而是能自动接到自己的判断系统里。

外部知识真正有价值的地方，不是“这篇文章讲了什么”，而是“它会如何改变我的判断和行动”。

## 3. AI AS ME 个人数字分身体系搭建、迭代与可视化

这是知识管理里最核心的一个场景。

我现在做的 AI AS ME，本质上是把我自己的一部分认知、经验、业务判断、表达偏好、客户背景和协作规则，沉淀成一个长期可调用的个人数字分身体系。

原来这些东西都在脑子里，或者散在各种文档里。AI 想要像我，就只能靠我临时说：你要像我一样判断，你要按我的风格写，你要知道我做企业 AI 落地。但这种说法太模糊。

现在我会让 Claude Code 帮我搭和迭代 AI AS ME。

输入可以是很多东西：我已有的 CLAUDE.md、个人知识库、余一或 Danilo 这类外部播客稿、我想新增的人脉与资源模块、复盘模块、社交帖子里提炼出来的个人原则。

输出也有几类。

第一类是新文件结构建议。比如要不要加 `6. 人脉与资源.md`，里面应该有哪些字段：关系来源、资源类型、可信度、可调用场景、注意事项等。

第二类是对标改造建议。比如读完别人怎么搭个人知识库，再对照我现在的结构，判断哪些模块可以升级，哪些没必要学。

第三类是 AI AS ME 的运行逻辑说明和 Mermaid 流程图。比如输入源从哪里来，怎么做决策分级，怎么处理红黄绿灯信号，怎么自我进化，信息怎么沉淀，最后输出到写作、销售、培训、交付哪些地方。

第四类是给别人复制这套体系的三步法。因为这件事后面也可以变成对外课程和咨询案例。

这件事的价值在于：AI AS ME 不再是一个模糊概念，而是一套可以被维护、解释和复制的系统。

它让我越来越清楚：知识库不是一堆资料，而是一个人的工作操作系统。

## 4. 知识库定向检索 / 主题聚合

知识库建起来之后，最常用的动作是检索。

但这里的检索，不是简单搜一个关键词。更多时候，我需要把散在多个文件里的内容，按某个主题重新聚合成一份报告。

比如我要见一个客户，需要把之前关于这个客户、这个行业、这个伙伴的所有记录捞出来；我要写一篇文章，需要把某个观点相关的案例、方法论、教训都找出来；我要参加一个活动，需要 AI 帮我从知识库里拼出一个对外 profile。

原来我只能自己在 Obsidian 里搜。搜出来很多文件，还要一个个打开，看哪段有用，再重新组织。

现在我会让 Claude Code 做定向检索。

输入是：指定目录和一批关键词。

输出是：按主题分块的聚合报告。里面会包括相关结论、真实案例、伙伴档案、经验教训，每条标来源文件。有些任务还会附现状诊断，比如数据反哺记录已经断了；有些会直接帮我写成对外叙事。

这个场景里，Claude Code 会先 glob 或 ls 列文件，再多关键词 grep 定位，最后批量 Read 核心文件，按主题归类。它也会纠正我给错的路径，不会硬凑结果。

这件事省心的地方是：散在十几个文件里的判断，可以一次性聚合成可用视图。

知识库如果只能存，不能快速捞出来，就只是仓库。能按任务重新聚合，它才是工作系统。

## 5. 大型素材库通读总结，生成结构化内容地图

有时候我面对的不是几十个文件，而是几千个文件。

比如公众号素材库、通往 AGI 之路、博主合集、整个内容创作库，可能有几千甚至 8500 多个文件。我不可能逐篇看，但我需要知道里面到底有什么，哪些有价值，哪些适合写作，哪些适合培训，哪些只是低价值存档。

原来这种盘点几乎做不了。文件太多，人工看不完；只看文件名，又看不出内容质量。

现在我会让 Claude Code 生成内容地图。

输入是：一个或多个素材目录。

输出是：逐篇或分组概要，加上整体分析。比如主题分布、内容质量、演变线、对业务的价值。它可以把公众号素材归纳成“个人成长 → 职场 → AI 工具入门 → AI 实操 → AI+出海”的演变线，也能识别不同号主的风格。

如果文件太多，Claude Code 会先读 `_index` 或目录结构拿全貌，再派多个 Agent 分目录并行处理。8500 个文件这种规模，就会拆成十几个子任务，让每个子任务负责一个部分。

这个场景的关键是：目标不是逐篇抄全文，而是做内容地图。

内容地图的价值是让我知道“我有什么”。很多人知识库里其实堆了很多东西，但没有地图，就等于没有。需要写文章、做课程、做白皮书时，还是不知道该去哪里找。

这件事省心的地方是：几千篇素材可以在短时间里变成一张可读的全景图。

## 6. 为《企业 AI 转型白皮书》建素材地图 / 章节素材调研

这个场景更具体，是为白皮书写作服务。

我在写企业 AI 转型白皮书时，手里有很多素材库：通往 AGI 的几千篇内容、公众号素材、培训材料、客户沟通、工作经历、面试文档等。但这些素材不可能全部直接用。我要知道哪些素材能支撑哪一章，哪些只是技术教程，可以跳过。

原来这件事很容易变成大海捞针。写某一章时临时找素材，效率低，而且容易漏掉好案例。

现在我会让 Claude Code 按章节框架做素材地图。

输入是：素材库目录、白皮书章节框架和提取重点。

输出是：一份素材地图。包括这个库的总览、和白皮书相关的部分、高价值内容清单、每条素材适合支撑哪一章、怎么用。如果是客户或项目类素材，还会输出项目客户清单、老板高频疑虑、框架调整建议。

这个场景里，Claude Code 不会逐篇读完几千篇，而是用 Glob 和 Grep 按关键词筛，比如企业落地、AI 转型、Agent、知识库、组织、行业案例。然后再判断哪些是真正可用案例，哪些是纯技术教程。

这件事省心的地方是：海量素材被重新按写作目标组织起来。

写白皮书不是把素材堆上去，而是把素材放到正确章节里。素材地图让写作不再靠临时搜索，而是有一套可复用的检索路径。

## 7. 飞书知识库整库/分模块批量迁移到本地 Obsidian

很多高价值资料在飞书里。

比如 WaytoAGI、求职知识、职场燃料风控、某些外部知识库。它们在线上能看，但要长期用于写作、培训和检索，我更希望把它们迁移到本地 Obsidian。

原来这类迁移非常麻烦。飞书知识库有层级，有文档，有表格，有权限，有些文档还会失效。手工复制几百篇几千篇文档，基本不现实。

现在我会让 Claude Code 批量迁移。

输入是：飞书 wiki 链接或知识空间 ID，再加筛选规则。比如只保留 2025-10-01 之后修改的内容。

输出是：本地按原多级层级还原的 Markdown 文件。每篇带 updated、source 等元数据。过期内容和空文件夹会删除，每个目录还会报告总篇数、删除数、保留数。

Claude Code 会用 `lark-wiki` 和飞书 OpenAPI 拿节点树和修改日期，再写脚本递归遍历下载 docx，按日期过滤。遇到 bitable 或 sheet 类型就跳过，遇到 token 失效就切到 web-access CDP 兜底。最后用 `find | wc -l` 自检，补漏。

这件事省心的地方是：巨型飞书知识库可以结构化搬到本地。

对我来说，这些知识库不是看一眼就完，而是后面写作、培训、咨询、白皮书的素材底座。迁移到本地之后，它们才能被 Claude Code 稳定读取和复用。

## 8. 飞书素材按主题归类填充进飞书知识库 wiki

有时候不是从飞书迁到本地，而是反过来：我要把本地已有素材整理进飞书知识库，方便团队查看。

比如我已经有几十到几百篇公众号素材，按 AI Agent、AI 行业、AIGC、AI 商业变现这些主题分好类。下一步是把它们变成飞书 wiki 里的结构化子页面。

原来这件事也很耗时间。要建子页、写主题概述、整理文章表格、复制摘要、放链接。几十篇还行，几百篇就非常累。

现在我会让 Claude Code 批量处理。

输入是：主题页 wiki_token、文章清单、本地原文目录、子页面划分规则。

输出是：飞书里建好的多个子页面。每页包括主题概述、完整文章表格、精选解读。比如 AI Agent 可以拆成框架概览、多 Agent 架构、落地案例；AIGC 可以拆成图像、视频、AI 写作。

Claude Code 会用 `lark-cli docs +create --wiki-node` 建子页，再用 Python 把标题匹配到本地 Markdown，提摘要，生成飞书 Markdown 表格，最后用 docs update 写入。

这件事省心的地方是：几百篇散文章可以自动变成团队可检索的飞书知识库。

这类动作适合团队协作。不是所有人都看我的 Obsidian，本地库更适合我自己用，飞书 wiki 更适合团队共享和客户交付。

## 9. 飞书文档读取/迁移/双向搬运/格式修复

飞书相关还有一类更通用的文档处理。

有时候我要快速读懂一个飞书 wiki 或文档；有时候要把飞书内容归档到 Obsidian；有时候要把本地报告写回飞书；有时候飞书表格显示乱了，需要修复；有时候两份简历或文档要融合重写。

原来这些事情都要打开飞书手动操作。复制出来的内容还常常带很多飞书标签，表格格式也容易乱。

现在我会让 Claude Code 做双向搬运和修复。

输入是：飞书链接和目标位置。

输出可能是：文档结构化梳理、去掉飞书标签的干净 Markdown、本地写回飞书的内容、修好的表格、去重融合后的文档。

Claude Code 会先用 `lark-wiki get_node` 把 wiki_token 换成真实 obj_token，再用 `lark-doc fetch` 拉正文。需要写回时，用 update 或 replace_range。遇到表格异常时，重写 Markdown 表格替换，并复验行列数。

这件事省心的地方是：飞书和本地知识库可以双向流动。

我不需要被某个平台锁住。飞书里的内容可以进本地，Obsidian 里的报告也可以回到飞书，格式问题也能顺手修掉。

## 10. 知识库素材资产化：提取案例、行业情报、方法论

知识库里最有价值的，不是所有信息，而是能反复复用的资产。

比如真实项目案例、行业情报、方法论、客户洞察、销售话术、工具对比、经验教训。这些东西如果只是藏在客户沟通记录里，就没有发挥最大价值。

原来我很难系统判断哪些文件值得沉淀。客户沟通很多，培训材料很多，项目记录很多，但不是每个都值得变成案例库或方法论。

现在我会让 Claude Code 做素材资产化评估。

输入是：若干文件夹路径和目标库格式。

输出是：每个文件有没有可提取的项目级案例、行业情报、方法论、客户洞察。它会给出“强烈推荐 / 推荐 / 无”的判断和理由，并建议文件名，比如 `YYYY-MM_行业_项目名_状态.md`。

更进一步，它还可以整理出方法论池、案例池、话术池、工具对比池。

这个场景里，我会要求它必须逐篇读内容，不能只看文件名。还要区分原创和第三方，每条标来源，找不到就写“原文未找到”，同时标明保密边界。

这件事省心的地方是：散落的项目记录可以被系统体检，判断哪些值得变成长期资产。

这一步非常重要。企业 AI 落地服务靠的不是单次项目，而是项目背后能沉淀出多少案例、方法论和可复用材料。

## 11. 知识库盘点统计：Token 量、文件数、规则库梳理精简

知识库越用越大，也会越来越重。

到一定阶段，我需要知道：这个知识库到底有多大？能不能一次加载进上下文？哪些目录最大？哪些文件最重？规则库是不是膨胀了？哪些规则应该合并，哪些高频铁律应该上移到 CLAUDE.md？

原来这些问题只能靠感觉。觉得知识库很大，但不知道具体多大；觉得规则很多，但不知道哪些重复。

现在我会让 Claude Code 做盘点统计。

输入是：某个目录或规则库。

输出是：文件数、字符数、Token 估算、目录占比、单文件排名。比如 AI AS ME 有 116 个文件、662KB、约 32 万字符、45 到 60 万 tokens。规则库则可以输出 37 个文件的逐个一句话摘要、类型、可合并组、合并后文件名、该上移的高频规则和冗余识别。

Claude Code 会用 find 统计文件数、字符数、目录大小，并按中文估算 token。规则库则逐个 Read，按摘要、合并、上移、冗余四块输出。

这件事省心的地方是：知识库维护从感觉变成数据。

知识库不是越大越好。太大、太散、太重复，AI 读取成本会变高，调用效果也会变差。定期盘点和精简，是让知识库保持可用的前提。

## 12. 文件/Skill/配置定位检索

这是一个特别实用的小场景。

我经常会记得自己装过某个 Skill，或者写过某个配置，但忘了名字和路径。比如语音识别到底是哪个 Skill？视频逐镜头解析是哪一个？终端自动保存会话那个 Skill 叫什么？某个客户材料放在哪个目录？

原来只能自己在文件夹里翻，或者凭关键词搜。文件多了以后很痛苦。

现在我会直接让 Claude Code 找。

输入是：一个模糊描述或关键词，再加候选目录。

输出是：精确文件路径清单和职责说明。

比如它会区分 `doubao-asr` 是语音识别，`content-video-parser` 是视觉理解；`claude-session-manager` 是终端关闭自动保存会话。它还会纠正一些语音识别导致的错误文件名，比如瑞图和瑞兔、林销易和林孝煜。

这件事省心的地方是：在庞大的本地知识库和 Skill 库里秒级定位能力。

它避免我重复开发，也避免我明明已经有工具却找不到。知识库和工具库越大，这个场景越有价值。

## 13. 经验/复利沉淀与价值提取

知识管理最有复利的部分，是把错误和洞察沉淀下来。

很多时候，Claude Code 做错了，被我指出来。如果只是当场改完，下一次还可能再犯。更好的做法是把这次错误写成规则。

同样，很多对话里会聊出一些很有价值的业务洞察，比如私教不等于培训，不等于陪跑；把价值讲透反而能把单做大；书可以变成授权背书资产；某个飞书格式坑以后要避免。这些如果不沉淀，很快就忘了。

现在我会直接说：“这个记一下”“沉淀一下”“这个不对”。

Claude Code 会自己判断该写到哪里：规则库、经验库、教训库、MEMORY.md、CLAUDE.md，或者某个业务方法论文件。

输入是：一次踩坑，或者一条洞察。

输出是：一条长期生效的防错规则，或者一条经验沉淀。比如语音转文本必须用豆包 ASR；技能评估要先看是否高频；飞书 Part 标题后要空行；PDF.js 内存里可以捞文件；售前推进要串成 SOP。

这件事省心的地方是：错误会变成系统改进，而不是一次性情绪。

Claude Code 不是每次都从零变聪明，而是通过规则库和经验库越来越少犯同类错误。这就是知识管理的复利。

## 14. 个人知识库重建工作叙事

有时候我不是要找某个资料，而是要让 AI 帮我重建一段经历。

比如我想总结最近半年干过什么，补一篇周记，回忆某期播客观点，查对公账户信息，梳理某个合伙人的沟通脉络。

原来这种事情靠自己回忆很费劲。尤其是过去半年做了很多项目、客户、培训、内容、工具，靠脑子想很容易漏。

现在我会让 Claude Code 从知识库里重建叙事。

输入可以很模糊：介绍我最近半年干过的事；帮我补本周周记；找一下之前某期播客讲过什么；查一下对公账户；梳理一下某个合伙人的历史沟通。

输出是：基于 MEMORY、经验库、教训库、会话记录、周记、每日晚报、播客目录等材料整理出来的工作回顾、周记文档、历史观点还原或关系脉络。

这个场景里，Claude Code 不一定能拿到原始全文，但可以从压缩记录里还原核心逻辑。比如某期播客原稿丢了，也可以从之前沉淀过的摘要里恢复主要观点。

这件事省心的地方是：知识库变成了外置记忆。

它不只是帮我存资料，还能帮我重新组织过去发生过的事，变成可对外讲、可复盘、可继续推进的叙事。

## 15. 业务工作流拆解到 XMind / 知识库问答智能体素材整理

最后一个场景，是把知识库用于更具体的系统化工作。

一类是把自己的业务流程外化成 XMind。比如获客、转化、交付，每个环节怎么走，哪些地方 AI 可以协助。原来这些流程都在脑子里，很难给别人讲，也很难优化。

现在我会让 Claude Code 根据我的口述和业务理解，生成分层工作模块清单，再输出 XMind 文件。比如 46 个节点、3 级结构，并标出 AI 协助环节。

另一类是把 Obsidian 里的客户沟通笔记整理成可导入扣子、Coze 这类知识库问答系统的素材。

原来客户沟通笔记可能分散在多个目录，文件名不统一，还有非 Markdown 文件。直接导入知识库，检索效果会很差。

现在我会让 Claude Code 整理目录：把所有 Markdown 文件拍平到根目录，重名加前缀，删除非 Markdown 和空目录，再给出分段建议。比如每条沟通用 `##` 独立分段，加客户、项目、日期这些元数据字段，提高后续检索粒度。

这件事省心的地方是：知识库不只是给人看的，也可以被整理成给智能体用的素材。

一个知识库如果想用于问答，不能只是文件堆。它要有分段、元数据、命名规则和边界。Claude Code 可以帮我把散乱资料整理成更适合向量化和检索的形态。

## 这类场景真正解决了什么问题

知识管理这一类场景，本质上解决的是“AI 怎么越来越懂我”的问题。

如果没有知识库，AI 每次都只能处理眼前任务。  
有了知识库，它才能知道我的业务、客户、案例、风格、规则、资源和判断标准。  
如果知识库还能持续更新，它就会从一次性工具变成长期工作系统。

这类场景里，Claude Code 的价值不是单纯存文件，而是把信息变成资产。

外部长文可以变成行为校准。  
客户沟通可以变成案例库。  
培训材料可以变成素材池。  
飞书知识库可以迁移到 Obsidian。  
Obsidian 内容可以反向写进飞书。  
错误可以变成防错规则。  
项目经验可以变成方法论。  
过去半年工作可以重建成叙事。  
散乱笔记可以整理成知识库问答素材。

这里有几个关键原则。

第一，知识库要服务任务。不是为了存而存，而是为了写作、销售、培训、交付、复盘时能快速调用。

第二，知识要有来源。案例、方法论、行业情报、客户洞察，都要能追溯到原文或原项目，否则后面容易失真。

第三，知识库要定期盘点。文件越多，越需要内容地图、Token 统计、规则精简和冗余清理。

第四，经验要及时沉淀。踩坑之后不写规则，下一次还会踩。业务洞察不写进经验库，很快就忘。

第五，知识库最终要变成工作流。AI AS ME、飞书迁移、Obsidian 检索、XMind 工作流、Coze 知识库素材整理，这些不是孤立动作，而是在搭一套长期可复用的工作系统。

## 下一篇写什么

这篇讲的是第十大类：知识管理。

下一篇可以继续拆第十一大类：自动化与视频产线。

如果说知识管理解决的是“怎么把信息沉淀下来并持续调用”，那自动化与视频产线解决的就是“怎么把内容、素材、截图、脚本、视频合成这些环节连成流水线”，让一条内容从 GitHub 项目、文章素材或视频参考，自动变成可发布的视频和图文。
