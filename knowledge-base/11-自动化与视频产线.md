# ClaudeCode 的 172 个应用场景（11）：自动化与视频产线


前面一篇讲的是知识管理。

知识管理解决的是：信息、经验、案例、规则怎么沉淀下来，并在下一次写作、销售、交付、培训时重新调用。

这一篇讲第十一大类：自动化与视频产线。

这个模块和前面几个模块不太一样。前面更多是在“处理内容”“处理资料”“处理客户”“处理文档”，这一篇更多是在讲：怎么把一连串重复动作做成自动化流水线。

比如看到一个 GitHub 项目，怎么自动变成短视频脚本？脚本和截图准备好以后，怎么自动合成 3:4 竖屏视频？视频合成失败了，怎么查后端、查截图、查字幕、查 TTS？每天工作结束后，怎么自动生成晚报？邮箱怎么每天自动汇总？Claude Code 的会话怎么自动存档？抖音收藏的视频怎么批量下载成本地素材库？

这些事情如果手工做，都不算特别难。但一旦每天做、批量做、持续做，就会变得非常耗时间。

Claude Code 在这里的价值，不是单次帮我完成某一步，而是把“输入 → 处理 → 输出 → 归档 → 复盘”这一整条链路串起来。

这一篇一共 11 个场景。

## 1. GitHub 项目转短视频：写脚本 + 中文化截图

这是视频产线的第一步：把一个 GitHub 项目变成一条短视频素材包。

原来如果我看到一个开源项目，想做成“逛逛 GitHub”风格短视频，需要自己读 README、看 Star、找亮点、写口播、截图、翻译页面、整理分镜。单个项目还行，如果一晚想连续做几十个项目，人工根本顶不住。

现在我会直接把 GitHub repo URL 丢给 Claude Code。

输入是：一个或多个 GitHub 项目链接，也可以是已经抓好的 `metadata.json` 和 README。

输出是：每个项目一个文件夹，里面有一份 `article.md` 和一组中文化截图。

`article.md` 不是普通文章，而是短视频分镜脚本。一般 4 到 6 个 `<!-- scene -->`，第一镜是 top 类型，后面是 section 类型。内容包括情绪开场、Star 数钩子、项目定位、核心亮点和收尾。每个分镜还有对应图片占位，比如 `![](images/001.png)`，并带截图注释。

截图部分会打开 GitHub 页面，把英文内容中文化，再根据分镜锚点精准截图。最后存到 `~/Desktop/抓取结果/{owner}_{repo}_{日期}/`。

这件事省心的地方是：从“发现一个项目”到“拿到一条短视频脚本 + 配套截图”，可以压到几分钟。

这一步的关键不是简单总结 GitHub，而是直接生成适合视频生产的中间产物。它给后面的自动合成做了标准化准备。

## 2. GitHub 脚本 + 截图合成 3:4 竖屏短视频

有了脚本和截图，还要变成真正的视频。

原来这一步要剪辑、配音、排字幕、对时间轴、加背景、加水印、导出视频。如果每个 GitHub 项目都这么做，效率会很低。

现在我会让 Claude Code 调本地视频产线自动合成。

输入是：一个包含 `article.md` 和 `images/` 的文件夹。

输出是：一条 3:4 竖屏 MP4，通常是 2160×2880 或类似规格。视频里有深蓝紫渐变背景，有“跟着曾俊学 AI”水印，有真人男声配音，有和语音对齐的字幕，也会按分镜切换截图。

比如一个 GitHub 项目，合成后可能是一条 31 到 50 秒、5 到 7MB 的短视频，直接存到 `~/Desktop/视频/`。

Claude Code 会调用 `github-video-composer` skill，再走本地 Pixelle-Video 后端，里面包括 FastAPI、TTS、Playwright 和 ffmpeg。

这件事省心的地方是：脚本、图、配音、字幕、背景、水印、导出全自动。

原来短视频最麻烦的是剪辑和字幕对轴。现在只要上游脚本格式稳定，后面合成就可以批量跑。一个项目一条视频，多项目就可以连续产出。

## 3. 视频合成后端 / 截图管线故障排查与修复

自动化产线最容易遇到的问题是：能跑一次，不代表能稳定跑一百次。

视频合成经常会出各种奇怪问题。比如截图中文化失败、后端 500、进程被 OOM 杀掉、视频时长显示异常、字幕和语音对不上、TTS 音色调用失败。

原来这种问题很烦。你只看到“合成失败”，但不知道是截图脚本的问题、翻译的问题、TTS 的问题、后端的问题，还是 ffmpeg 的问题。

现在我会让 Claude Code 深入排查代码和日志。

输入是：失败现象和报错日志。

输出是：根因定位和修复方案。

比如截图中文化失败，原因可能是 CSS 注入触发 React 重渲染，把翻译后的内容冲掉。解决方法是不要用容易变动的文字做锚点，而是改用稳定子串，比如 `v2.0`、`(67)`、emoji 这类不会被翻译改动的内容。

比如 TTS 报 500，Claude Code 会去读 `tts_service.py`，发现 local 模式其实只走豆包，edge-tts 从没被真正用上，于是修改逻辑，让 Neural 音色直连 edge-tts。

比如 OOM，就降分辨率；比如 PTS 时间戳没重置，播放器把视频读成 33 分钟，就加 ffmpeg 重编码修复。

这件事省心的地方是：视频产线的问题不再靠玄学试错，而是能回到源码和日志里定位。

自动化最重要的不是“跑通一次”，而是“失败了知道怎么修”。Claude Code 在这里就像一个产线工程师，帮我把偶发问题变成可修复、可沉淀的规则。

## 4. GitHub 视频脚本批量质检评分 / 改稿前后对比

视频脚本批量生成之后，还有一个问题：质量不稳定。

有些脚本开头有钩子，有些像说明书；有些分镜完整，有些图片占位缺失；有些语言口语化，有些 AI 味很重；有些项目亮点没讲出来，有些只是复述 README。

原来我只能一篇篇看。几十篇脚本看下来，非常累，而且标准也容易飘。

现在我会让 Claude Code 批量质检。

输入是：抓取结果目录，比如 36 个 `article.md`，再加评分规则。规则可以包括：有没有 `<!-- scene -->`、有没有截图注释、有没有图片占位、分镜数够不够、字数是否合适、开头钩子强不强、口语化够不够、有没有 AI 味。

输出是：按评分降序的总览表、Top 3 最好脚本、Top 5 最差脚本，以及每篇的问题点。

如果是改稿前后对比，还可以输入旧版和新版，让 Claude Code 按 7 条标准评分，输出旧分、新分、升幅、主要改进和还可以优化的地方。

这件事省心的地方是：几十篇脚本不用人工逐篇读完，先由 Claude Code 统一打分筛选。

更重要的是，改稿有没有用可以量化。不是凭感觉说“新版好像更好”，而是能看到平均分提升了多少，具体提升在哪。

## 5. GitHub 项目写作素材简报提取

有时候我不想让 AI 直接写脚本，只想让它帮我备料。

比如我想自己写一条视频脚本，但不想先读完整 README。那就可以让 Claude Code 先把项目变成素材简报。

输入是：5 个项目文件夹，每个里面有 `metadata.json` 和 README，或者至少有 `article.md` 兜底。

输出是：每个项目一份“写作素材简报”。

简报里会包括：项目名、Star 数、作者背景、项目定位、最大亮点、真实用户或媒体提及，以及这个项目更适合哪种写作公式。比如适合故事型、数据型、问题型还是金句型。

这个场景的重点是：只提取素材，不代写。

原来我需要自己读 README 找亮点，现在 Claude Code 先帮我把信息压成一张素材卡。我再决定怎么写、用什么语气、怎么开头。

这件事省心的地方是：保留人的表达判断，同时减少读资料的时间。

很多时候，AI 直接写出来不一定像我。但让它先整理素材，再由我自己写，会更稳。

## 6. 视频生成工具开发与迭代

视频产线不是一开始就能用。

一开始可能界面丑、字幕对不上、TTS 不自然、清晰度不够、片尾没有转化动作、工具还不能复用。要真正变成可用产线，需要连续打磨。

原来这些优化都要自己改代码、调前端、调后端、调字幕、调 TTS。很多细节不大，但堆起来很耗时间。

现在我会让 Claude Code 直接参与工具开发和迭代。

输入是：各种体感问题。

比如：

- 前端太丑，但功能不要变
- 字幕要两行滚动，时间轴要和语音对齐
- SEO、API、AI、HTML、GPT 这些缩写读音不对
- 想接入豆包或火山引擎 TTS
- 视频清晰度从 3:4 升到 2K
- 片尾要加点赞、关注、转发、头像和微信号
- 最后封装成可复用 skill
  
输出是：前端改版、后端修复、字幕逻辑调整、TTS 读音映射、音色接入、清晰度提升、片尾 CTA、以及 `article-to-video` 或 `video-auto-pipeline` skill。

这个场景里，Claude Code 的工作方式是：先复述需求确认，再读代码，尽量保持原功能不变，只改体验问题。改完后用预览服务器验证，再继续修 bug。

这件事省心的地方是：视频工具可以从“能跑的脚本”打磨成“可持续使用的产线”。

自动化工具真正有价值，不是第一次能生成视频，而是能稳定生成、效果能看、风格统一、遇到问题能修、最后能封装成 skill 反复调用。

## 7. 每日工作晚报自动生成

这个场景和视频没关系，但属于自动化体系里非常重要的一块。

我每天和 Claude Code 做很多事：客户沟通、培训材料、文章写作、项目方案、知识库整理、工具调试。如果不复盘，很多工作痕迹会散掉。

原来写日报要手动回忆：今天做了什么，有哪些产出，有什么决策，哪些没做完，明天要干什么。每天写太麻烦，不写又容易丢。

现在我搭了一套每日工作晚报自动生成机制。

输入来自四类数据：

- 当天全部 Claude Code 会话内容
- 当天 Obsidian 修改文件清单
- 命令历史
- 当日 GitHub Trending
  
输出是一份 `# 晚报 YYYY-MM-DD`。

内容包括：GitHub 值得关注项目、当天一句话总结、按项目分组的关键产出、重要决策、遗留问题、金句和灵感、明日待办、数据源统计。

它会按 cwd 判断项目归属。比如日常沟通归客户沟通，企业培训归培训，Obsidian 和 AI AS ME 归知识库。GitHub Trending 还会把英文描述翻成中文，过滤 fork 和低价值项目。

这件事省心的地方是：每天的工作痕迹自动沉淀成结构化复盘。

我不需要晚上手写日报，也不会因为忙而漏掉重要进展。长期看，这些晚报本身也会变成个人工作档案。

## 8. 每日邮件汇总定时推送

除了工作晚报，邮箱也可以自动化。

原来每天打开邮箱，要从很多营销邮件、通知邮件里找真正重要的内容。合同、报价、会议、客户回复，可能夹在一堆噪音里。

现在我会让 Claude Code 做每日邮件汇总。

输入是：近 7 天 Gmail 邮件，后来调整成每天定时只抓 Gmail。

输出是一份邮件汇总。里面包括概览、按发件人和日期归类的表格、重点关注项，并判断有没有合同、报价、会议、紧急事项。营销邮件会被自动过滤掉。

定时任务可以设置为每天 9:50 自动推送。

这件事省心的地方是：每天不用翻邮箱，先看一份摘要。

它不是替我处理邮件，而是帮我先筛一遍，把真正需要注意的事情提出来。

## 9. 定时任务体系搭建 / 盘点 / 改时间

自动化任务一多，就会出现另一个问题：任务到底跑在哪里？

比如每日晚报为什么关了 Claude Code 还会推？邮件汇总在哪里设置的？到底是 macOS 的 launchd，还是 Claude Code 内置 Cron？要改时间，应该改哪个文件？今天漏推了，怎么手动补？

原来这种问题很容易混乱。尤其是多套自动化同时存在时，如果没有盘点，后面维护会很麻烦。

现在我会让 Claude Code 帮我查定时任务体系。

输入是：一个问题，比如“为什么关终端还会推送”“帮我把日报推一下”“把邮件汇总改时间”。

输出是：完整体系说明和操作结果。

比如它会查到：每日晚报是 macOS LaunchAgent 在 22:33 调 `daily-report.sh`，和 Claude Code 是否打开无关；邮件汇总是 Claude Code 内置 Cron；相关配置可能在 `scheduled_tasks.json`、crontab、plist 里。

Claude Code 会用 `find`、`grep`、`launchctl`、`plutil`、CronList 等方式检查状态。需要改时间就改时间，需要手动触发就手动触发。

这件事省心的地方是：自动化系统不再是黑盒。

自动化越多，越需要知道每个任务在哪里、谁触发、什么时候跑、失败了怎么查。否则一旦出问题，就会变成“它怎么又没跑”的玄学。

## 10. 会话存储管理：按项目归档 + 全量存档 + 导出备份

Claude Code 本身的会话也需要管理。

原来我最担心的是：终端关掉之后，历史上下文丢了。一个项目聊了很多轮，里面有客户信息、方案判断、决策记录、代码修改、经验教训。如果不归档，下次很难接上。

现在我会让 Claude Code 把会话按项目存到 Obsidian。

输入是：一句话，比如“把这次会话存到会话某某项目”。

输出是：Obsidian `会话/项目名/` 下的文件夹，里面包括 context、记忆压缩、决策记录、会话记录。关终端时，还会自动覆盖更新为完整版。

后来我还设计了一整套项目会话管理机制：目录结构、模板、记忆压缩、同步逻辑、SessionEnd hook 和信号处理器。

更大的动作是全量导出备份。

Claude Code 帮我定位到两层存储：一层是 claude-code-sessions 元数据，里面有 UI 备注 title；另一层是 `~/.claude/projects` 里的 jsonl 全量原文。最后写 Python 脚本，把 781 个 jsonl 一次性复制到桌面，按备注或内容命名，重名加序号。

这件事省心的地方是：Claude Code 的历史会话不再只是聊天记录，而是可以变成长期资产。

这也是我后来能做“172 个应用场景提取”的前提。没有全量会话导出，就没有后面的场景库。

## 11. 抖音收藏 / 链接批量下载 + 取消收藏

最后一个场景，是短视频素材自动化。

我经常会在抖音收藏一些素材视频，后面想做 TikTok 素材库、选题库或视频参考。但如果只是收藏在抖音里，很难管理，也很难批量分析。

原来如果要整理收藏夹，就要一个个点开、复制链接、下载视频，再手动取消收藏。几十个视频就很麻烦。

现在我会让 Claude Code 走浏览器登录态批量处理。

输入是：抖音收藏夹。

输出是：本地 `~/Desktop/TikTok素材/` 下的 MP4 文件，比如 38 个视频全部下载成功，0 失败。下载完成后，再批量取消收藏。

Claude Code 会用 `chrome-cdp` 连接已登录抖音的 Chrome，先读 DOM 看收藏数，滚动加载全部视频，抓取全部链接。然后用 `tiktok-scraper` 写批量下载脚本。串行太慢就改成 5 路并行，下载完再比对缺失，只补没下成功的。最后进入抖音批量管理，全选取消收藏。

这件事省心的地方是：收藏夹不再是平台里的临时仓库，而是可以变成本地素材库。

更重要的是，它把下载和清理连成一条线。素材拿到本地后，抖音收藏夹也顺手清空，避免越积越乱。

## 这类场景真正解决了什么问题

自动化与视频产线这一类场景，本质上解决的是“重复工作怎么流水线化”的问题。

看到一个 GitHub 项目，不再是手动读 README、写脚本、截图、剪辑。  
一批脚本写完，不再是人工逐篇质检。  
视频合成失败，不再是凭感觉重试，而是读日志、查代码、修后端。  
每天工作结束，不再手动写晚报。  
每天早上，不再手动翻邮件。  
定时任务不再黑盒，而是能查、能改、能补推。  
Claude Code 会话不再散落，而是能按项目归档。  
抖音收藏不再留在平台里，而是变成本地素材库。

Claude Code 在这里的价值，不是某个单点能力，而是把很多原本分散的动作接起来。

它可以读 GitHub。  
可以写短视频脚本。  
可以截图。  
可以合成视频。  
可以排查后端。  
可以批量评分。  
可以开发工具。  
可以每天自动生成晚报。  
可以定时汇总邮件。  
可以管理定时任务。  
可以归档历史会话。  
可以批量下载素材。

这里有几个关键原则。

第一，自动化要先有稳定中间格式。比如 GitHub 短视频产线里，`article.md` 和 `images/` 就是中间格式。上游稳定，下游才能自动合成。

第二，跑通一次不等于产线成熟。真正的产线要能处理报错、失败、OOM、字幕错位、TTS 问题、截图失败这些异常。

第三，自动化要可维护。定时任务跑在哪里、谁触发、日志在哪、怎么手动补推，都要说清楚。否则自动化会变成新的黑盒。

第四，批量任务要有质检。脚本批量生成之后，要评分；视频批量合成之后，要检查；邮件汇总要过滤噪音；晚报要按项目归类。

第五，历史记录本身也是资产。会话存储、晚报、素材库、脚本评分、排错规则，这些都不是一次性产物，后面还可以继续反哺知识库和内容创作。
