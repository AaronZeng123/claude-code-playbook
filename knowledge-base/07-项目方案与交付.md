# ClaudeCode 的 172 个应用场景（7）：项目方案与交付


前面一篇讲的是客户沟通与销售。

客户沟通解决的是：怎么判断客户、怎么理解需求、怎么报价、怎么推进合作。

但客户点头之后，问题还没有结束。真正难的是：怎么把一句“我们想做个智能体”，翻译成一个能评审、能报价、能开发、能交付的方案。

很多企业 AI 项目，失败不是因为模型不行，而是前面方案没讲清楚。

客户说“想做一个 AI 系统”，但到底输入什么、输出什么、数据从哪里来、谁来用、怎么判断效果、哪些功能一期做、哪些功能先不做，很多时候并没有被定义清楚。

这一篇讲第七大类：项目方案与交付。

这个模块一共 6 个场景：

1. 智能体/AI 落地方案设计  
2. 竞品 SaaS/方案文档拆解 + 推断客户真实意图  
3. 系统流程图 / 泳道图 / 甘特图绘制  
4. 多 Agent 协作架构图绘制并嵌入飞书  
5. AI 电商商品图生成系统方案设计  
6. 客户语音 + 图片需求翻译与可行性判断
  
它们共同解决一个问题：把客户脑子里模糊的需求，变成可讨论、可裁剪、可交付的方案。

## 1. 智能体/AI 落地方案设计

这是项目方案里最核心的场景。

客户经常会说：“我们想做一个智能体。”但这句话本身没法交付。

什么叫智能体？是一个聊天机器人？是一个知识库问答？是一个能调用系统接口的工作流？是一个推荐系统？还是一个自动生成结果并回填表格的业务助手？

原来如果客户这么说，我需要自己在脑子里拆：这个业务的输入是什么，输出是什么，数据在哪里，用户是谁，AI 负责哪一段，人负责哪一段，边界在哪里。拆完之后，还要再写成客户能看懂的方案。

现在我会让 Claude Code 帮我把客户需求翻译成智能体设计方案。

输入是：业务场景、客户需求要点、已有沟通记录、客户最关心的能力点。

输出是：一份更像系统设计的方案。里面会包括：

- 入参是什么
- 出参是什么
- 用户怎么发起任务
- 系统怎么做意图识别
- 什么时候走知识库检索
- 什么时候走推荐逻辑
- 什么时候引导用户继续补充信息
- 最后结果以什么形式输出
- 数据从哪里来
- 哪些能力一期做
- 哪些能力暂时不做
  
比如点餐推荐智能体，可以拆成用户输入、API 网关、意图识别、菜单知识库、推荐策略、库存和高毛利商品优先级、最终推荐结果。

再比如社交媒体监测方案，可以拆成采集层、存储层、分析层和展示层。采集层负责拿数据，存储层可能用 ES，分析层用 LLM 和业务 Skill，展示层可能是 Dashboard，结果以 API 形式输出。

这件事省心的地方是：客户原本只说“做个智能体”，Claude Code 可以帮我拆成可评审的系统方案。

它不是直接写代码，而是先把项目定义清楚。

对企业项目来说，这一步非常关键。因为客户只有认可“系统怎么跑”，后面才好谈价格、排期和交付范围。否则双方都在说智能体，但理解可能完全不同。

## 2. 竞品 SaaS/方案文档拆解 + 推断客户真实意图

有时候客户不会直接说需求，而是发来一份产品手册、竞品方案或帮助文档，然后问：“这个能不能做？”

这种场景很容易误判。

客户表面上是在问功能，实际可能是在问：能不能替代这个产品？能不能比这个便宜？能不能帮我降低内部结算成本？能不能只做其中几个关键模块？

原来我拿到这类文档，要自己一页页看。比如一份科研助手手册、一个实验工坊 PDF、一个竞品 SaaS 产品介绍。看完之后还要判断：这个产品本质上做什么，哪些功能复杂，哪些依赖外部 API，哪些功能其实可以砍掉，客户真正想让我做的是补充还是替代。

现在我会把这类文档交给 Claude Code 拆。

输入是：竞品产品手册、方案 PDF、docx 帮助文档，外加客户原始需求。

输出通常包括几部分。

第一部分是竞品功能拆解。Claude Code 会把对方产品拆成几个模块，逐个说明功能本质。

第二部分是和客户原需求的覆盖对照表。哪些需求被竞品覆盖了，哪些没有覆盖，哪些只是看起来相关。

第三部分是客户真实意图推测。比如客户可能不是想补充一个小工具，而是想替代现有产品；可能不是想做全套，而是想用更低成本实现其中几块能力。

第四部分是技术裁剪建议。比如哪些模块依赖外部 API，哪些模块成本高，哪些可以先砍掉，哪些可以找可替代的开源数据源，比如 arXiv、CORE、OpenAlex、NSTL。

这件事省心的地方是：客户发来的不是一个“文档”，而是一堆商业信号。

Claude Code 可以帮我把这些信号拆出来，判断这个项目到底该不该做、能不能做、报价天花板在哪里、哪些模块不应该接。

这个场景很适合科研智能体、SaaS 替代、竞品复刻、客户已有系统改造这类项目。客户问的是“能不能做”，我需要回答的是“哪些能做，哪些不建议做，怎么裁剪才合理”。

## 3. 系统流程图 / 泳道图 / 甘特图绘制

很多项目方案光靠文字讲不清楚。

客户看一大段系统说明，很容易看晕。特别是智能体项目，里面有用户、系统、知识库、模型、接口、后台、人工审核多个角色，如果不画图，客户很难理解系统到底怎么跑。

原来我需要自己画流程图、泳道图、甘特图。要么用 PPT 手画，要么用画图工具拖节点。画一张还好，改几轮就很烦。

现在我会让 Claude Code 直接根据方案生成图。

输入是：系统设计、项目周期、业务流程逻辑，或者交付流程说明。

输出可以是：

- Mermaid 系统运行流程图
- 项目甘特图
- 交付泳道图
- 售前判断流程图
- 通用架构运行图
  
比如点餐智能体，可以画成：用户输入 → API 网关 → 意图识别 → 检索/推荐分支 → 结果生成 → 用户确认。

再比如 Agent 定制交付，可以画成客户、曾俊、技术合作方三条泳道，包含需求确认、方案设计、原型开发、测试反馈、上线交付等阶段。

如果 Mermaid 画出来太丑，或者客户场景需要更好看的图，也可以换成飞书 UML 画板或 SVG 重绘。Claude Code 可以先生成 Mermaid，保证逻辑能跑通，再根据需要改成更适合放进飞书文档或方案 PPT 的版本。

这件事省心的地方是：方案可视化不用从零手画。

更重要的是，画图过程会反过来检查方案逻辑。流程图画不出来，往往说明方案本身还没拆清楚。比如谁先触发、哪个系统取数、模型什么时候介入、人工在哪里审核，这些问题只有画图时才会暴露。

## 4. 多 Agent 协作架构图绘制并嵌入飞书

有些方案或案例需要更专业的架构图。

比如一个质量审核项目、一个多 Agent 协作方案，单纯文字讲“识别、检索、生成、回填”不够有说服力。客户看不到系统结构，也就不容易相信你真的想清楚了。

原来我如果要做架构图，需要自己设计图形、画模块、调布局、导出图片，再手动插入飞书或 PPT。这个过程很耗时间，而且一不小心就做得很粗糙。

现在我会让 Claude Code 直接生成多 Agent 协作架构图。

输入通常只是一句话：“帮我绘制一张架构图。”再加上项目背景和系统逻辑。

输出是一套本地产物，包括 SVG、PNG、JSON，有时还会直接嵌入飞书文档末尾。

架构图一般会分成几个模块：

- 输入层：用户需求、业务文件、表格、图片、系统数据
- 多 Agent 流水线：识别 Agent、检索 Agent、生成 Agent、回填 Agent
- 知识库治理：文档解析、切片、标签、索引、更新、权限
- 评估反馈回路：人工校验、结果修正、日志沉淀、持续优化
  
Claude Code 会先建目录写 SVG，再渲染检查 PNG，必要时导出 JSON。写入飞书前还会 dry-run 探测，避免直接把文档弄坏。确认没问题后，再用飞书工具把白板或图片嵌进去。

这件事省心的地方是：原本纯文字的方案或案例，可以很快补上一张专业架构图。

对客户来说，架构图的作用不是装饰，而是增强可信度。尤其是多 Agent、知识库、评估反馈这些概念，如果只靠文字讲，会显得虚；一旦画成结构图，客户就更容易理解你在交付什么。

## 5. AI 电商商品图生成系统方案设计

这个场景来自电商商品图生成需求。

客户可能是卖袜子、服饰或其他电商商品的商家，希望用 AI 批量生成商品图：参考图风格生成、颜色合集图、卖点图、批量生成等等。

原来拿到这种需求，很容易直接想“做个网页工具”。但网页不一定是最合适的技术形态。客户真正需要的可能不是一个完整 SaaS，而是一个能读取素材、套用规则、批量生成图片的工作流或 Skill。

现在我会让 Claude Code 先做技术形态判断和 MVP 路径设计。

输入是：项目背景、客户已有素材、参考图风格、商品白底图、颜色图、卖点文案、期望生成的图片类型，以及模型能力边界，比如 GPT Image 2。

输出包括：

- 技术形态建议
- Skill 和网页的适用边界
- MVP 路径
- 商家素材结构分析
- 具体输入输出设计
- 核心用例
- 待确认问题
  
比如 Claude Code 会判断：如果客户现在只是需要一套可控的批量生成流程，Skill 可能比网页更合适。因为网页涉及账号、权限、前端、后端、存储、队列、部署，成本更高。Skill 更适合先跑通商家素材结构和生成规则，等流程稳定后再考虑产品化。

它还会帮我重新定义输入输出。

输入不是一句“生成商品图”，而是：白底商品图、参考风格图、颜色列表、卖点文案、目标平台尺寸、输出图片类型。

输出也不是一张图，而可能是：单品主图、颜色合集图、卖点说明图、场景图、批量生成文件夹。

这件事省心的地方是：它能帮我避免过度工程。

很多项目一上来就做网页，反而会把简单需求复杂化。先判断技术形态，再设计 MVP，能降低交付风险。

## 6. 客户语音 + 图片需求翻译与可行性判断

还有一种很常见的顾问式场景：客户发来一段语音、一张截图，或者一份看起来很宏大的 AI 规划文档，让你帮他看看。

这类需求往往很模糊。

客户可能自己也没想清楚。他发来的可能是 AI 生成的一大套规划，看起来很完整，有很多模块，有 600 多行内容，但真正能落地的部分可能很少。

原来我需要自己听语音、看截图、理解客户真正想问什么，再判断这个方案是不是过度设计。

现在我会让 Claude Code 帮我做“需求翻译”。

输入是：客户语音转文字、截图、客户当前阶段、上下文背景。

输出是四层判断：

- 他做了什么
- 他想要你做什么
- 他真正卡在哪里
- 你应该提醒他什么
  
比如有客户发来一份 613 行 AI 规划文档，看上去很宏大。但 Claude Code 读完后可能会判断：这份文档大概率是 AI 生成的，严重超出客户当前阶段。客户真正需要的不是继续扩展方案，而是做减法，先抓 P0 任务，优先找零成本、低风险、能快速见效的点。

这个场景里还有一个细节：语音转写可能会错。比如客户说的是 BI，转写成了 V1。如果不结合上下文，很容易误解需求。Claude Code 会根据业务语境纠正这类误识别。

这件事省心的地方是：客户发来的模糊信息，可以被翻译成清晰判断。

对顾问来说，这非常重要。客户经常不是缺方案，而是方案太多、太大、太虚。真正有价值的不是继续帮他加功能，而是帮他做减法，判断哪些先做、哪些暂时别做。

## 这类场景真正解决了什么问题

项目方案与交付这一类场景，本质上解决的是“从需求到交付物”的翻译问题。

客户讲的是业务语言：我想做智能体，我想做 AI 系统，我想生成商品图，我想替代某个 SaaS，我想把这份规划落地。

但交付需要的是另一套语言：输入、输出、数据源、运行逻辑、系统边界、MVP、流程图、架构图、模块拆分、交付周期、技术裁剪。

Claude Code 的价值，就是帮我在这两套语言之间做转换。

一句“做个智能体”，要变成入参、出参、运行逻辑和系统架构。  
一份竞品手册，要变成功能拆解、报价判断和裁剪建议。  
一段客户语音，要变成真实需求和可行性判断。  
一个电商图片需求，要变成 Skill 方案和 MVP 路径。  
一段复杂流程，要变成流程图、泳道图、甘特图。  
一份文字案例，要补上多 Agent 协作架构图。

这里有几个关键原则。

第一，先定义边界，再谈交付。AI 项目最怕边界模糊。数据从哪里来，结果输出到哪里，哪些系统要接，哪些先不接，必须提前讲清楚。

第二，方案不能只写文字。流程图、架构图、泳道图、甘特图能让客户更快理解，也能暴露方案里没想清楚的地方。

第三，MVP 优先于大而全。客户一开始经常想要很多功能，但真正能交付、能验证价值的，通常是更小的路径。先做 P0，再扩展。

第四，技术形态要慎重判断。不是所有需求都要做网页，也不是所有需求都要做智能体。有些用 Skill 更合适，有些用脚本更合适，有些只需要先做咨询判断。

第五，顾问价值在于做减法。客户给你一份很宏大的规划时，不一定要顺着继续扩。很多时候真正有价值的是帮客户砍掉不该做的东西，找到第一步。
